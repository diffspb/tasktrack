# 07. Процесс принятия решения при мульти-назначении

Это уникальный механизм продукта, который отличает его от классических таск-трекеров. Он активируется, когда у задачи **несколько исполнителей**, и определяет, как задача финально закрывается.

## Суть

В обычном таск-трекере закрытие задачи = исполнитель поставил статус «Сделано». Здесь иначе:

> Когда несколько исполнителей работают над одной задачей — каждый независимо приходит к своему решению. Задача не закрывается автоматически. Вместо этого запускается **процесс принятия решения**: решения презентуются, оцениваются, и уполномоченное лицо выбирает итог.

## Жизненный цикл задачи с несколькими исполнителями

```
Задача создана
    ↓
Исполнители назначены (с ролями)
    ↓
Каждый ведёт свой воркфлоу независимо
    ↓
Каждый презентует своё решение (переходит в финальный статус)
    ↓
[Процесс принятия решения]
    ├─ Decision-maker просматривает все решения
    ├─ Может отправить одно / несколько / все на доработку
    └─ Выбирает итог: одно решение или несколько
    ↓
Задача закрыта. Итоговая резолюция = выбранное решение
```

## Критерии принятия решения

Критерии задаются прямо в задаче при её создании (или редактировании). Примеры:

- «Минимальное время выполнения»
- «Наименьшая стоимость реализации»
- «Максимальный охват тест-кейсов»
- Произвольный текстовый критерий

Отбор по критериям **может быть автоматизирован** — если критерий формализуем и решение содержит измеримые данные, система может предложить победителя автоматически. Финальный выбор остаётся за decision-maker'ом.

## Роли в процессе

| Роль | Кто | Что делает |
|------|-----|------------|
| **Исполнитель** | Назначен на задачу | Выполняет работу, презентует решение |
| **Decision-maker** | Заказчик задачи или делегат | Принимает финальное решение |
| **Ревьюер** (опционально) | Назначен менеджером | Может оценивать решения, но не выбирает итог |

Decision-maker назначается явно при создании задачи. По умолчанию — автор задачи.

## Возможные исходы процесса

1. **Принято одно решение** — стандартный случай. Задача закрыта, победившее решение зафиксировано.
2. **Принято несколько решений** — гипотеза для тестирования. Например, два подхода берутся в реализацию параллельно или применяются в разных контекстах.
3. **Отправлено на доработку** — decision-maker указывает, какие решения (одно / несколько / все) нужно доработать и почему. Затронутые исполнители возвращаются в рабочий воркфлоу.

## Влияние на модель данных

Ключевые сущности, которые появляются из этого механизма:

- **Assignment** — запись о назначении: `task_id`, `user_id`, `role`, `personal_status`, `workflow_id`
- **Solution** (решение исполнителя) — `assignment_id`, `content`, `submitted_at`, `status` (draft / submitted / accepted / revision_requested / rejected)
- **DecisionCriteria** — критерии на уровне задачи: `task_id`, `description`, `is_automated`
- **TaskDecision** — итоговое решение: `task_id`, `decision_maker_id`, `accepted_solution_ids[]`, `decided_at`, `note`
