# 07. Процесс принятия решения при мульти-назначении

> **MVP-упрощение (актуальное состояние кода):** В текущей реализации таблицы `Assignment`, `Solution` и `TaskDecision` отсутствуют. Задача имеет одного исполнителя (`Task.assignee_id`). Суррогат Decision Process: комментарий с `labels=["solution"]` маркирует Solution, `meta.solution_comment_id` хранит ссылку. Полноценный Decision Process (несколько исполнителей с независимыми воркфлоу, API submit/make-decision) — см. `docs/tech-debt.md`, реализуется после MVP-запуска.
>
> Раздел ниже описывает **целевую механику** (v2/post-MVP). Используйте как продуктовое требование при проектировании восстановления Assignment и Decision Process.

Это уникальный механизм продукта, который отличает его от классических таск-трекеров. Он активируется, когда у задачи **несколько исполнителей**, и определяет, как задача финально закрывается.

> 📖 Терминология: ниже строго разводятся **Solution** (поданное решение исполнителя) и **Decision** (итоговое решение decision-maker'а). См. [glossary.md](./glossary.md).

## Суть

В обычном таск-трекере закрытие задачи = исполнитель поставил статус «Сделано». Здесь иначе:

> Когда несколько исполнителей работают над одной задачей — каждый независимо приходит к своему **Solution** (поданному решению). Задача не закрывается автоматически. Вместо этого запускается **процесс принятия решения** (Decision Process): Solution'ы презентуются, оцениваются, и decision-maker выносит **Decision** — выбирает один или несколько Solution как итог.

## Жизненный цикл задачи с несколькими исполнителями

```
Задача создана
    ↓
Исполнители назначены (с ролями)
    ↓
Каждый ведёт свой воркфлоу независимо
    ↓
Каждый подаёт Solution (переходит в финальный статус)
    ↓
[Decision Process]
    ├─ Decision-maker просматривает все Solution
    ├─ Может отправить одно / несколько / все на доработку
    └─ Выносит Decision: принимает один или несколько Solution
    ↓
Задача закрыта. Итоговая резолюция = выбранный Decision
```

## Критерии принятия решения

Критерии задаются прямо в задаче при её создании (или редактировании). Примеры:

- «Минимальное время выполнения»
- «Наименьшая стоимость реализации»
- «Максимальный охват тест-кейсов»
- Произвольный текстовый критерий

Отбор по критериям **может быть автоматизирован** — если критерий формализуем и Solution содержит измеримые данные, система может предложить победителя автоматически. Финальный Decision остаётся за decision-maker'ом.

## Роли в процессе

| Роль | Кто | Что делает |
|------|-----|------------|
| **Исполнитель** | Назначен на задачу | Выполняет работу, подаёт Solution |
| **Decision-maker** | Заказчик задачи | Выносит Decision |
| **Ревьюер** (опционально) | Назначен менеджером | Может оценивать Solution'ы, но не выносит Decision |

Decision-maker назначается явно при создании задачи. По умолчанию — автор задачи. Делегирование роли decision-maker'а — 🟡 v2.

## Возможные исходы процесса

1. **Принят один Solution** — стандартный случай. Задача закрыта, победивший Solution зафиксирован как Decision.
2. **Принято несколько Solution** — гипотеза для тестирования. Например, два подхода берутся в реализацию параллельно или применяются в разных контекстах.
3. **Отправлено на доработку** — decision-maker указывает, какие Solution (один / несколько / все) нужно доработать и почему. Затронутые исполнители возвращаются в рабочий воркфлоу.

## Влияние на модель данных

Ключевые сущности, которые появляются из этого механизма:

- **Assignment** — запись о назначении: `task_id`, `user_id`, `role`, `personal_status`, `workflow_id`
- **Solution** (поданное решение исполнителя) — `assignment_id`, `content`, `submitted_at`, `status` (draft / submitted / accepted / revision_requested / rejected)
- **DecisionCriteria** — критерии на уровне задачи: `task_id`, `description`, `is_automated`
- **TaskDecision** — итоговое решение: `task_id`, `decision_maker_id`, `accepted_solution_ids[]`, `decided_at`, `note`

## Структура Solution

Solution — это запись о предложенном варианте исполнителя. В MVP:

- **Содержимое:** свободный текст (форматированный) + необязательные вложения (файлы до 20 МБ или URL-ссылки).
- **Версионирование:** отсутствует. Исполнитель может редактировать Solution в статусе `draft` и после отзыва. Предыдущие версии не хранятся явно — история изменений фиксируется в аудит-логе.
- **Подача:** явное действие («Подать Solution»). До подачи — `draft`, доступен только исполнителю.
- **Отзыв:** исполнитель может отозвать поданный Solution (вернуть в `draft`), пока decision-maker не начал рассмотрение (пока нет ни одного перехода в `accepted` / `revision_requested`).

## Структура DecisionCriteria

- **Формат:** упорядоченный список текстовых критериев на задаче.
- **Необязательны:** задача может не содержать критериев.
- **Редактирование:** задаются при создании задачи или редактировании. Блокируются для изменения, как только хотя бы один исполнитель подал Solution.
- **Автоматизация:** нет в MVP. Отбор по критериям — только ручной.

## Видимость Solution'ов

Режим по умолчанию — **открытый**: все участники задачи (исполнители, decision-maker, ревьюеры, наблюдатели) видят все поданные Solution'ы в любой момент.

Слепой режим (Solution'ы скрыты до вынесения Decision) — в v2.

## Состояния задачи и состояния Solution

### Глобальные статусы задачи

```
open
  → in_progress          (хотя бы один исполнитель начал работу)
  → awaiting_decision    (все исполнители подали Solution)
  ↔ in_revision          (decision-maker отправил ≥ 1 Solution на доработку)
  → decided              (Decision вынесен)
  → closed               (задача явно закрыта)
```

`decided` и `closed` — два **отдельных видимых статуса**:
- `decided` = Decision вынесен, но задача ещё формально не закрыта. Участники могут ознакомиться с Decision.
- `closed` = задача закрыта явным действием decision-maker'а или менеджера (переход `decided → closed` — не автоматический; требует подтверждения).

Для задач с одним исполнителем: `open → in_progress → closed` (Decision Process не запускается).

### Состояния Solution

```
draft → submitted → accepted
                 → revision_requested → (исполнитель дорабатывает) → submitted
```

### Переходы задачи в `awaiting_decision`

Задача переходит в `awaiting_decision` автоматически, когда все **lead**-исполнители перевели свои Solution в `submitted`.

## SLA и уведомления

- Жёсткого дедлайна на Decision Process нет в MVP.
- При переходе задачи в `awaiting_decision`: decision-maker получает уведомление (in-app + email).
- Напоминание decision-maker'у: через 3 дня, если Decision не вынесен. Период настраивается на уровне проекта (0 = без напоминаний).
- Эскалации (передача задачи другому decision-maker'у) — v2.

## Цикл доработки

1. Decision-maker выбирает один или несколько Solution, которые нужно доработать, и пишет текстовый комментарий с обратной связью.
2. Задача переходит в `in_revision`. Затронутые исполнители получают уведомление.
3. Каждый затронутый исполнитель видит комментарий к своему Solution, редактирует его и повторно подаёт.
4. Когда все Solution из статуса `revision_requested` снова поданы → задача автоматически возвращается в `awaiting_decision`.
5. Количество циклов доработки не ограничено.
6. Исполнитель, чей Solution не был отправлен на доработку, остаётся в статусе `submitted` — повторно подавать не нужно.

## Несколько принятых Solution

По умолчанию Decision-maker выбирает один победивший Solution.

Режим **multi-accept** включается флагом при создании задачи («допускается несколько принятых Solution'ов»). В этом режиме decision-maker может принять несколько Solution одновременно; все они фиксируются как Decision.

Семантика: «берём несколько подходов в реализацию» или «применяем в разных контекстах». В MVP — поддерживается, но UI должен явно показывать, что это исключение, а не правило.
