## 1. Противоречия и несоответствия

### 1.1. Статусы задачи: `07-decision-process.md` vs `14-diagrams.md`

**Где:** `07-decision-process.md` (раздел «Глобальные статусы задачи») vs state-диаграмма в `14-diagrams.md`.

**Конфликт:** В `07-decision-process.md` схема статусов задачи не включает прямой переход `open → closed` при одном исполнителе — там написано «open → in_progress → closed». Однако в state-диаграмме `14-diagrams.md` переход `decided → closed` помечен как «Автоматически (или менеджер)», что создаёт неясность: автоматически означает немедленно, или это двухшаговый процесс? Нет ни одного перехода `open → in_progress → closed` в диаграмме для случая одного исполнителя — только примечание. При этом ERD в `12-data-model.md` не содержит поля `global_status` в таблице `Task` — есть только `current_status_id` (ссылка на `Status` в воркфлоу). Поле `global_status` используется в API-ответах, но в ERD отсутствует.

**Рекомендация:** Добавить поле `global_status enum` в таблицу `Task` в ERD `12-data-model.md`. Уточнить в `14-diagrams.md` переход `decided → closed`: либо это автоматический немедленный триггер (тогда `decided` — транзитный статус, не нужный пользователю), либо это отдельный шаг (тогда описать, кто и когда его совершает).

### 1.4. Роль `decision_maker` в матрице прав и ERD

**Где:** `13-permissions.md` (раздел «Decision Process: права decision-maker'а») vs `12-data-model.md`.

**Конфликт:** В матрице прав decision-maker описан как «отдельная роль на задаче, не AssigneeRole». В ERD поле `decision_maker_id` хранится в таблице `Task`. При этом матрица прав проверяет ProjectRole и AssigneeRole — а decision-maker не является ни одним из них. Нет описания, как именно проверяется право «вынести Decision»: через сравнение `task.decision_maker_id == current_user_id` или через какой-то другой механизм. Это оставляет реализацию на усмотрение разработчика, что риск.

**Рекомендация:** Добавить в `13-permissions.md` явное описание проверки прав decision-maker'а: «проверяется `task.decision_maker_id == current_user.id` или, если делегирован, соответствующая запись делегирования». Механизм делегирования вообще нигде не описан в ERD — в таблице `Task` нет поля для делегирования.

### 1.6. API не соответствует ERD: нет `global_status` в модели данных

**Где:** `15-api.md` (ответы GET /tasks/{id}, GET /projects/{id}/tasks) vs `12-data-model.md`.

**Конфликт:** API повсеместно возвращает поле `global_status` (строки: `"open"`, `"in_progress"`, `"awaiting_decision"` и т.д.), однако в ERD таблицы `Task` такого поля нет. Есть `current_status_id` — FK на таблицу `Status` воркфлоу (персональный статус задачи в воркфлоу). Это разные вещи: глобальный статус (`open/in_progress/awaiting_decision/...`) не является статусом воркфлоу.

**Рекомендация:** Добавить поле `global_status enum('open','in_progress','awaiting_decision','in_revision','decided','closed')` в таблицу `Task` в ERD. Это поле меняется триггерами (логикой бэкенда) при изменении состояния Assignment'ов и Solution'ов.

### 1.9. Роли в `02-roles.md` не совпадают с ролями в `13-permissions.md`

**Где:** `02-roles.md` vs `13-permissions.md`.

**Несоответствие:** `02-roles.md` перечисляет роли: Автор (Reporter), Исполнитель (Assignee), Наблюдатель (Watcher), Менеджер проекта / Тимлид, Администратор, Стейкхолдер/заказчик. ## 2. Риски реализуемости

### Decision Process (state machine задачи с мульти-Assignment'ами) — 🔴 Высокий риск

Глобальный статус задачи (`open/in_progress/awaiting_decision/in_revision/decided/closed`) — производное состояние, вычисляемое из состояния множества Assignment'ов и Solution'ов. Логика:
- Переход в `awaiting_decision`: все lead-Assignment'ы в финальном статусе воркфлоу **И** все их Solution в `submitted`.
- Переход в `in_revision`: decision-maker отправил ≥1 Solution на доработку.
- Возврат в `awaiting_decision`: все Solution из `revision_requested` снова в `submitted`.

Это нетривиальная state machine с несколькими параллельными дорожками. Риски:
- Race condition: два lead-исполнителя подают Solution одновременно → оба проверяют условие «все lead'ы подали» до того, как транзакция другого зафиксируется → двойной переход.
- Конкурентное редактирование задачи при одновременном изменении нескольких Assignment'ов.
- «Удалённый lead с submitted Solution» создаёт логику пересчёта агрегированного состояния.

**Рекомендация:** Реализовать переходы глобального статуса через транзакционный SELECT FOR UPDATE на строку задачи при каждом изменении Assignment/Solution. Весь пересчёт глобального статуса — одна транзакция.

---

### Персональный воркфлоу (Assignment.current_status_id) — 🟡 Нетривиально

Два независимых измерения статуса: `Task.global_status` и `Assignment.current_status_id` (N штук). Kanban-доска должна отображать задачу в колонке по персональному статусу пользователя (а не глобальному). Фильтрация «мои задачи в статусе X» должна фильтровать по персональному статусу, а не глобальному.

Это потребует нестандартных JOIN'ов при выборке задач: `JOIN assignments ON task.id = assignment.task_id AND assignment.user_id = :current_user`. При масштабе 30 000 задач и 300 пользователей запросы нужно тщательно индексировать.

**Рекомендация:** Индексы: `(task_id, user_id)` на `Assignment`, `(user_id, current_status_id)` на `Assignment`.

---

### Матрица прав (две ортогональные системы) — 🟡 Нетривиально

ProjectRole × AssigneeRole × `decision_maker_id` — три источника прав. Реализация проверки прав в middleware или сервисном слое должна учитывать все три. Edge cases:
- Пользователь одновременно является decision-maker и lead-исполнителем — может и работать, и выносить Decision (документация не запрещает).
- Пользователь с ProjectRole=Viewer, который является lead через AssigneeRole — по матрице `13-permissions.md` он получает право «Вести персональный воркфлоу», несмотря на Viewer.
- Кросс-проектные связи задач: нужен read-доступ к обоим проектам — значит права проверяются в контексте нескольких проектов одновременно.

**Рекомендация:** Реализовать отдельный сервис/модуль проверки прав с явным покрытием тестами для всех edge cases из матрицы. Не встраивать проверку прав в контроллеры напрямую.

---

### Оптимистичные блокировки — 🟡 Нетривиально

`Task.version` защищает от конфликтов при редактировании задачи. Однако при мульти-Assignment'ах каждый lead-исполнитель меняет свой персональный статус, что тоже инкрементирует версию задачи (если логика бэкенда так устроена). Это означает, что два исполнителя, работающих одновременно, будут постоянно получать 409. 

Если смена статуса Assignment не трогает `Task.version` — тогда возникает вопрос: какие именно операции инкрементируют версию? Документация это не уточняет.

**Рекомендация:** Явно определить, какие операции инкрементируют `Task.version`: только редактирование полей задачи (заголовок, приоритет, описание), но не смена Assignment.current_status_id. Это снимет ложные конфликты.

---

### Soft-delete с сохранением ссылок — 🟡 Нетривиально

Документация описывает soft-delete для `Task` и `Project`. Однако возникают сценарии:
- Удалённый пользователь (soft-delete не описан для User!) — что происходит с его Assignment'ами, Solution'ами, TaskDecision'ами?
- `TaskDecision.accepted_solution_ids[]` — массив UUID без FK-ограничений (в PostgreSQL массив не поддерживает FK). Если Solution физически удалён — ссылка становится «битой».
- `Comment.author_id` при удалении пользователя — FK нарушается.

В ERD не определён soft-delete для `User` вообще, хотя поле `is_active` присутствует.

**Рекомендация:** Определить политику для `User`: вместо удаления — деактивация (`is_active = false`). Физическое удаление пользователей не поддерживать. Это снимает проблему битых FK. Добавить описание в `12-data-model.md`.

---

### Полнотекстовый поиск (PostgreSQL tsvector) — 🟢 Стандартно для MVP-масштаба

При 30 000 задач и поиске по заголовку + описанию + комментариям `tsvector` на PostgreSQL — правильный выбор. Комментарии потребуют `tsvector` на отдельной таблице и отдельного индекса. Для русскоязычного контента нужна правильная конфигурация `text search config` (`russian`) — иначе стемминг не работает.

**Риск:** В `12-data-model.md` схема поиска не описана: нет поля `search_vector` в таблице `Task` или `Comment`, нет упоминания триггеров для обновления `tsvector`. Без этого разработчик может забыть реализовать поиск правильно.

**Рекомендация:** Добавить в `12-data-model.md` описание: вычисляемое поле `search_vector tsvector` в `Task` (объединение `title || description`) и `Comment`, обновляемое триггером. Конфигурация: `russian`.

---

### Схема воркфлоу как снимок или ссылка в Assignment — 🔴 Высокий риск

В `12-data-model.md` написано: Assignment хранит `workflow_id` — ссылку на `Workflow`. Если менеджер меняет воркфлоу проекта (добавляет/удаляет статусы, меняет переходы), то незакрытые Assignment'ы продолжают ссылаться на изменённый воркфлоу. Это может привести к:
- Исполнитель находится в статусе, которого больше нет в воркфлоу.
- Допустимые переходы изменились, и исполнитель застрял.

В документации написано: «Изменения воркфлоу применяются к новым задачам; существующие задачи остаются в текущем статусе» — но это правило для задачи, не для воркфлоу. Если сам воркфлоу модифицируется (а не клонируется), все существующие Assignment'ы по этому воркфлоу затронуты.

**Рекомендация:** Зафиксировать правило: воркфлоу иммутабелен после появления хотя бы одного Assignment, ссылающегося на него. Изменение воркфлоу = создание нового воркфлоу и опциональная миграция открытых задач. Это стандартная практика (Jira так и делает).

---

### Конкурентная подача двух Solution одновременно — 🟡 Нетривиально

Если два lead-исполнителя нажимают «Подать Solution» почти одновременно, оба запроса `POST /solutions/{id}/submit` могут одновременно проверить условие «все lead'ы подали» и оба инициировать переход в `awaiting_decision`. В PostgreSQL без явной блокировки это гонка.

**Рекомендация:** При обработке `submit` выполнять `SELECT FOR UPDATE` на `Task` перед проверкой условия перехода. Тогда второй запрос будет ждать завершения первого.

---

### Хранение `accepted_solution_ids[]` как PostgreSQL-массива — 🟡 Нетривиально

`TaskDecision.accepted_solution_ids uuid[]` — денормализованное поле без FK-ограничений в PostgreSQL. Это осознанное решение, описанное в `12-data-model.md`. Рисков немного (Decision иммутабелен, Solution не удаляются физически), но проверка «существует ли Solution с таким UUID» должна быть на уровне приложения, не БД.

**Рекомендация:** Добавить CHECK-constraint или application-level валидацию перед созданием `TaskDecision`.

## 3. Валидация API

### 3.1. Полнота: покрытие MVP-сценариев

**Отсутствующие эндпоинты:**

| Эндпоинт | Зачем нужен |
|----------|-------------|
| `PATCH /tasks/{task_id}/decision-maker` | Делегирование DecisionMaker (заявлено в матрице прав как MVP) |
| `GET /users` или `GET /projects/{id}/members/search` | Поиск пользователей при назначении исполнителя и приглашении в проект (упоминается в AC, но эндпоинта нет) |
| `PATCH /projects/{project_id}/workflows/{workflow_id}` | Обновление воркфлоу (переименование) |
| `DELETE /projects/{project_id}/workflows/{workflow_id}/statuses/{status_id}` | Удаление статуса из воркфлоу |
| `DELETE /projects/{project_id}/workflows/{workflow_id}/transitions/{transition_id}` | Удаление перехода |
| `GET /users/me/dashboard` | Дашборд «Мои задачи» (ключевой экран, нет эндпоинта) |
| `PATCH /users/me` | Обновление профиля (имя, аватар, часовой пояс) |
| `GET /projects/{project_id}/resolutions` | Список резолюций проекта |
| `POST /projects/{project_id}/resolutions` | Создание резолюции |
| `PATCH /resolutions/{resolution_id}` | Обновление резолюции |
| `GET /auth/oauth/google` | Инициация Google OAuth (есть в user stories A, нет в API) |
| `GET /projects/{project_id}/tasks/{task_id}` (или редирект) | Задача известна по ключу (`BACK-42`) — нет GET по ключу задачи |
| `DELETE /tasks/{task_id}/links/{link_id}` | Удаление связи между задачами |
| `GET /tasks/{task_id}/links` | Список связей задачи |
| `POST /tasks/{task_id}/labels` | Добавить метку к задаче |
| `DELETE /tasks/{task_id}/labels/{label_id}` | Удалить метку с задачи |
| `GET /projects/{project_id}/tasks/export` | Есть, но не описан формат полей CSV |

**Вывод:** API достаточно для прохождения happy path S1–S5, но недостаточен для полноценной работы UI — отсутствуют эндпоинты управления профилем, дашборда, операций с резолюциями и частичного управления воркфлоу.

---

### 3.2. Консистентность

**Именование — несоответствия:**

- `POST /solutions/{id}/revision` — использует слово `revision` как действие. Параллельные эндпоинты `POST /solutions/{id}/submit` и `POST /solutions/{id}/withdraw` следуют паттерну «глагол действия». `revision` — не глагол. Лучше: `POST /solutions/{id}/request-revision` или `POST /solutions/{id}/revise`.
- Параметр в `PATCH /assignments/{assignment_id}`: тело может содержать либо `status_id`, либо `assignee_role`. Это два разных действия в одном эндпоинте. По REST лучше разделить: `PATCH /assignments/{id}/status` и `PATCH /assignments/{id}/role`.
- В `POST /tasks/{task_id}/decision-criteria` параметр называется `items`, в `GET /tasks/{task_id}/decision-criteria` ответ тоже `items` — согласовано. Но в `PATCH /tasks/{task_id}/decision-criteria` описано «перезаписать весь список» — это поведение PUT, не PATCH. Следует использовать PUT.

**Пагинация:** Все списки используют `total + items` — единообразно. Хорошо.

**Формат ошибок:** Единый формат `{ "error": { "code", "message", "details" } }` — хорошо. Но конкретные error codes (строки типа `TASK_NOT_FOUND`) не перечислены как полный справочник — разработчик будет придумывать их самостоятельно.

**Рекомендация:** Добавить приложение «Справочник error codes» в `15-api.md`. Переименовать `POST /solutions/{id}/revision` в `POST /solutions/{id}/request-revision`.

### 3.3. Edge cases, не покрытые в API

| Ситуация | Что происходит (не описано) |
|----------|-----------------------------|
| `GET /tasks/{id}` для soft-deleted задачи | 404 или 410 Gone? Нет описания |
| `GET /tasks/{id}` для задачи из недоступного проекта | 403 или 404? (Утечка информации: 403 говорит о существовании) |
| `POST /tasks/{id}/decisions` при concurrent submit | Что если два решения отправлены одновременно? 409 только для второго, но оба могут получить 201 |
| `POST /solutions/{id}/submit` когда задача уже в `awaiting_decision` | Что если исполнитель подаёт вторично после отзыва? Поведение не описано |
| `PATCH /tasks/{id}` без `version` | Ошибка или принимается? Документ неоднозначен |
| `DELETE /assignments/{id}` для lead с `submitted` Solution | Ошибка 400 описана, но не описано: что происходит с Solution? |

### 3.4. Безопасность

**Потенциальные утечки данных:**

1. **Consultant и чужие Solution:** `GET /tasks/{task_id}/solutions` возвращает `403 Forbidden — нет прав (consultant до Decision)`. Однако `GET /assignments/{assignment_id}/solution` проверяет права по-другому. Если consultant знает `assignment_id` коллеги, может ли он получить его Solution напрямую? Явной проверки нет.

2. **Видимость email в ответах API:** `GET /projects/{project_id}/members` возвращает `email` участника. Viewer-уровень пользователь видит emails всех участников проекта — это может быть нежелательно.

3. **`GET /projects` для restricted-проекта:** Не описано явно, что пользователь без доступа получает при обращении к `GET /projects/{id}` для restricted-проекта — 403 или 404. 404 безопаснее (не раскрывает факт существования).

4. **`GET /tasks/{task_id}/decisions` доступность:** Нет явного описания прав. Любой Viewer может прочитать Decision? Это нормально, но стоит зафиксировать.

5. **Soft-deleted задачи в поиске:** `GET /search` не описывает исключение soft-deleted задач явно (хотя предполагается).

**Рекомендация:** Добавить в `15-api.md` раздел «Политика доступа к ресурсам» с явным описанием: что возвращается для soft-deleted объектов, как ведут себя restricted-проекты (404 vs 403), кто видит email участников.

## 4. Архитектурные стратегии

### Стратегия A: Monolith-first (рекомендуется)

**Описание:** Классический монолит: один бэкенд-сервис, один фронтенд-SPA, PostgreSQL, файловое хранилище для вложений.

**Стек:**
- Бэкенд: Python (FastAPI) или Go (Chi/Echo), PostgreSQL 16, Redis (очередь email + сессии)
- Фронтенд: React + Vite или SvelteKit
- Email: SMTP через Redis-очередь (Celery/rq для Python, или просто goroutine с каналом для Go)
- Файлы: локальный диск с nginx для отдачи (достаточно для 500 МБ / проект)
- Деплой: Docker Compose (web + db + redis + worker)

**Даёт:**
- Самый быстрый старт: один репозиторий, одна кодовая база.
- Простая отладка сложной state machine Decision Process — всё в одном процессе.
- Транзакции PostgreSQL покрывают всю бизнес-логику без распределённых транзакций.
- Достаточно для 50–300 пользователей без оптимизаций.

**Теряет:**
- При росте потребует горизонтального масштабирования — тогда нужен рефакторинг или выделение сервисов.
- Если в будущем понадобится публичный API и вебхуки — придётся добавлять rate limiting и очереди поверх.

**Когда выбирать:** Один разработчик или команда 2–3 человека. Исследовательский запуск. Нет требований к 99.9% uptime.

---

### Стратегия B: Monolith + Background worker

**Описание:** Та же монолитная архитектура, но с явным выделением фоновых задач в отдельный процесс-воркер с Redis-очередью. Воркер отвечает за: отправку email, вычисление `tsvector` (можно сделать через триггеры), напоминания decision-maker'у.

**Стек:**
- То же, что стратегия A, + явный воркер (Celery для Python / Go-воркер)
- Docker Compose: web + db + redis + worker (4 контейнера)

**Даёт:**
- Email-отправка не блокирует HTTP-ответ.
- Напоминания (через 3 дня без Decision) реализуются через scheduled tasks.
- Воркер можно масштабировать независимо при росте нагрузки на уведомления.

**Теряет:**
- Сложнее отлаживать распределённые сценарии: ошибка в воркере не видна в HTTP-ответе.
- Добавляет зависимость от Redis.

**Когда выбирать:** Когда надёжность email критична для Decision Process (а в данном продукте она критична — email уведомляет decision-maker'а). Рекомендуется с первого дня, если email в MVP.

---

### Стратегия C: Django/Rails + встроенная ORM

**Описание:** Full-stack фреймворк с батарейками: встроенная ORM, миграции, admin-панель (очень полезна для внутреннего инструмента), встроенная аутентификация.

**Стек:**
- Django REST Framework (Python) + PostgreSQL + Celery + Redis + React SPA
- Или Ruby on Rails API + Sidekiq

**Даёт:**
- Django admin = готовая панель для физического удаления, управления пользователями, аудита — без отдельной разработки.
- Встроенная система миграций.
- Богатая экосистема для типичных задач (OAuth через `django-allauth`, поиск через `django-haystack`).

**Теряет:**
- Django REST Framework медленнее FastAPI при высокой нагрузке (не критично для 50 пользователей).
- Rails — нишевой выбор, найти разработчиков сложнее.
- ORM может скрывать неэффективные запросы при сложных JOIN'ах (персональный статус Assignment).

**Когда выбирать:** Команда с Django/Rails-опытом. Нужна быстрая admin-панель без отдельной разработки.

---

### Рекомендация

**Оптимальный вариант для первого запуска: Стратегия B** (Monolith + Background worker).

Обоснование:
- Email для Decision Process — не опциональная фича, а часть ключевого сценария S2. Его надёжная доставка требует очереди с повторными попытками.
- State machine Decision Process — нетривиальная логика, которую проще всего отлаживать в одном процессе (монолит), а не в распределённой системе.
- Docker Compose с 4 контейнерами (web + db + redis + worker) — стандартная конфигурация, понятная любому разработчику.
- PostgreSQL покрывает все требования: tsvector, оптимистичные блокировки, транзакции для state machine.
- Масштаб 50→300 пользователей не требует горизонтального масштабирования в обозримой перспективе.

Выбор языка: Python/FastAPI если команда Python; Go если нужна производительность и команда Go. Принципиальной разницы при данном масштабе нет.

## 5. Открытые архитектурные вопросы

Вопросы, требующие ответа до начала разработки:

1. **Механизм делегирования DecisionMaker:** Как именно пользователь делегирует роль — через смену `task.decision_maker_id` или через отдельную таблицу истории делегирований? Нужна ли история (кто был decision-maker до делегирования)?

2. **Иммутабельность воркфлоу:** Можно ли редактировать воркфлоу, пока на нём есть активные задачи? Если нет — нужен UI-запрет и описание в API (сейчас нет ни того, ни другого).

3. **`global_status` — поле или вычисляемое значение:** Хранить `Task.global_status` как поле в БД (денормализация, нужны триггеры) или вычислять на лету из агрегации Assignment'ов? При масштабе MVP — хранить проще, но нужна гарантия консистентности.

4. **Soft-delete для User:** Что происходит с аккаунтом пользователя, если администратор хочет его удалить? Деактивация (`is_active = false`) или что-то иное? Что видят другие пользователи вместо его имени в задачах?

5. **Авторизация cross-project TaskLink:** Для установки связи между задачами разных проектов нужен read-доступ к обоим проектам. Как это проверяется на уровне API: один эндпоинт с проверкой обоих проектов? Нет описания в `15-api.md`.

6. **tsvector и русский язык:** Какая text search configuration используется: `russian` (требует установки snowball), `simple` (без стемминга)? Выбор влияет на качество поиска с первого дня.

7. **Канбан-доска с персональным статусом:** Если пользователь имеет несколько Assignment'ов в разных задачах, в какой колонке отображается задача на общей доске проекта — по глобальному статусу или персональному? AC в `stories/decision-process.md` говорит «в режиме "Мои задачи"» — но как работает общая доска проекта?

8. **Аватары пользователей:** Где хранятся? Локальный диск (проще для Docker Compose) или S3-совместимое хранилище (лучше для масштабирования)? Лимит 2 МБ на аватар указан в AC, но не в NFR.

9. **Concurrent revision cycles:** Если decision-maker отправил Solution на доработку, исполнитель его повторно подал, но decision-maker снова хочет отправить на доработку — количество циклов не ограничено. Нет описания, есть ли верхний лимит или предупреждение.

10. **Транзакционность перехода `decided → closed`:** Это два отдельных статуса или один? Если два — кто и когда совершает переход `decided → closed`? Если автоматически — это триггер в БД или логика бэкенда?

## 6. Итоговая оценка готовности

### Сильные стороны документации

- **Глоссарий** — образцовый. Строгое разведение Solution/Decision, canonical EN-names для кода — реально экономит время при разработке.
- **Decision Process** (`07-decision-process.md`) — полностью описан: состояния, переходы, revision-цикл, SLA, структуры данных. Один из самых проработанных разделов.
- **ERD** (`12-data-model.md`) — покрывает все основные сущности, содержит обоснование нетривиальных решений (денормализация accepted_solution_ids, soft-delete, оптимистичные блокировки).
- **Матрица прав** (`13-permissions.md`) — детальная, покрывает пересечение ProjectRole × AssigneeRole × decision-maker.
- **MVP-приоритизация** (`09-mvp.md`) — чёткая, 5 сценариев с обоснованием. Хорошая основа для итерационной разработки.
- **NFR** (`10-nfr.md`) — реалистичны для масштаба. Нет завышенных требований.

### Слабые стороны

- **Пропущен `global_status` в ERD** — критическое несоответствие между моделью данных и API. Разработчик сразу наткнётся на это.
- **Делегирование DecisionMaker** — заявлено как MVP в матрице прав, но нет ни эндпоинта, ни модели данных, ни AC.
- **Управление воркфлоу — неполный API** — нет эндпоинтов удаления статусов и переходов, нет правила об иммутабельности воркфлоу при активных задачах.
- **Отсутствует дашборд "Мои задачи"** в API — ключевой экран упомянут в MVP-приоритизации (🟢), но эндпоинта нет.
- **Права на `GET /tasks/{task_id}/decisions` и аналогичные read-эндпоинты** нигде явно не описаны.
- **Политика soft-delete для User не определена** — создаёт риск при реализации.

### Готовность к передаче в разработку

**Вердикт: условно готов.** Пакет документации достаточен для начала разработки ядра (S1, S2, S3 сценарии), но перед стартом необходимо:

1. Добавить `global_status` в ERD — 30 минут работы аналитика.
2. Определить судьбу делегирования DecisionMaker (MVP или v2) — 1 час.
3. Добавить эндпоинты дашборда, профиля, резолюций в `15-api.md` — 2–3 часа.
4. Закрыть вопрос об иммутабельности воркфлоу — 30 минут.
5. Зафиксировать политику soft-delete для User.

После этих исправлений пакет полностью готов к передаче в разработку без дополнительных уточнений по основным сценариям.
