# 06. Следующие шаги

> **Актуальный рабочий план — в [08-pre-design-tasks.md](./08-pre-design-tasks.md).**
> Этот файл оставлен как историческая roadmap-зарисовка раннего этапа.

## Что делать дальше

1. **Ревью и приоритизация историй из [03-user-stories.md](./03-user-stories.md).**
   Пройдись по разделам A–Q и отметь:
   - 🟢 нужно в MVP,
   - 🟡 нужно, но позже,
   - 🔴 не нужно вообще,
   - ✏️ требует изменения формулировки.

2. **Ответы на открытые вопросы в [04-multi-assignee.md](./04-multi-assignee.md).**
   Это блокер для проектирования модели данных. Без явных ответов модель `Assignment` будет переделываться 2–3 раза.

3. **Формулировка MVP-границы.**
   Минимальная версия, которая уже даёт ценность. Кандидат на формулировку:
   > Можно создать проект, создать задачу, назначить на двоих исполнителей, каждый проводит её по фиксированному воркфлоу из 3–4 статусов, при закрытии выбирает резолюцию из фиксированного списка, задача автоматически закрывается по правилу «все закрыли», итоговая резолюция выбирается автоматически по приоритету.
   Всё, что сверх — следующие итерации.

4. **Дополнение списка из [05-often-missed.md](./05-often-missed.md).**
   Решить, что из этого закладываем в архитектуру с первого дня, а что откладываем.

5. **Только после этого — переход к моделированию домена.**
   Сущности, связи, агрегаты, доменные сервисы. До этого момента код писать рано.

## Чек-лист готовности к проектированию

- [ ] Все истории из [03-user-stories.md](./03-user-stories.md) промаркированы.
- [ ] Все открытые вопросы из [04-multi-assignee.md](./04-multi-assignee.md) закрыты.
- [ ] Сформулирован MVP с конкретными пользовательскими сценариями (5–10 штук).
- [ ] Принято решение по пунктам из [05-often-missed.md](./05-often-missed.md).
- [ ] Выбран технологический стек (язык, фреймворк, БД).
- [ ] Заведён внешний трекер задач для самого проекта (пока свой не готов).

## Предлагаемый формат ведения проекта дальше

- Документация в этом репозитории растёт вместе с продуктом.
- Решения по архитектуре фиксировать как ADR (Architecture Decision Records) в отдельном каталоге `/docs/decisions/`.
- Истории из этого файла переезжают в трекер по мере приоритизации.
- Эти markdown-файлы остаются как «продуктовый референс» — словарь, к которому возвращаемся при обсуждении новых фич.
