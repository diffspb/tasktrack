# 11. Задачи для системного аналитика

Рабочий документ. Аналитик получает продуктовую документацию (`00`–`10`) как входные данные и должен произвести следующие артефакты.

**Входные данные:**
- Продуктовый контекст и границы → `00-context.md`
- Словарь терминов → `glossary.md`
- User stories (домены A–T, ~87 историй) → `03-user-stories.md`
- Ключевая фича и роли исполнителей → `04-multi-assignee.md`
- Decision Process: состояния, Solution, revision → `07-decision-process.md`
- MVP-сценарии и приоритизация → `09-mvp.md`
- NFR → `10-nfr.md`

Задачи сгруппированы по блокам. Рекомендуемый порядок: 0 → 1 → 2 → 3 → 4 → 5.

---

## Блок 0. Рефакторинг документации user stories

> Этот блок — первый, потому что его результат (новая структура историй) становится входом для блока 1 (AC).

### 0.1. Переработать структуру документов user stories

- **Зачем:** сейчас все ~87 историй лежат в одном файле `03-user-stories.md`, плюс часть историй — в `04-multi-assignee.md`. Когда к каждой 🟢-истории добавятся AC, файл станет нечитаемым. Аналитику неудобно работать с ним как с рабочим документом, а не как со справочником.
- **Варианты** (выбрать один):
  - **A. Разбить по группам доменов** — 4–5 файлов: `stories-core.md` (A–C, F–H), `stories-collaboration.md` (I, N, M), `stories-management.md` (D, E, G, K, L, Q), `stories-access.md` (P, R, S), `stories-decision-process.md` (T + истории из 04). Компромисс между размером и количеством файлов.
  - **B. Один файл на домен** — 20 файлов вида `stories-A-auth.md`, `stories-T-decision-process.md`. Удобно для глубокой работы с отдельным доменом; неудобно для быстрого обзора.
  - **C. Один файл на историю** — максимальная детализация, хорошо для тикетного трекера, тяжело для навигации без инструментов.
- **DoD:** выбран и зафиксирован вариант структуры; истории перенесены в новую структуру; `03-user-stories.md` либо становится индексом с оглавлением, либо удаляется; `README.md` обновлён.

### 0.2. Согласовать пересечение `03` и `04`

- **Зачем:** истории о мульти-исполнителях дублируются между `03-user-stories.md` (раздел T, F) и `04-multi-assignee.md`. После рефакторинга должен быть один канонический источник.
- **DoD:** в `04-multi-assignee.md` нет самостоятельных user stories — только продуктовые решения и описание механики. Все истории — в структуре из задачи 0.1.

## Блок 1. Acceptance Criteria для MVP-историй

### 1.1. Написать AC для всех 🟢-историй

- **Зачем:** без AC невозможно оценить, готова ли фича. Разработчик будет интерпретировать историю по-своему.
- **Формат:** Given / When / Then или маркированный чек-лист условий — на усмотрение аналитика, но единообразно.
- **DoD:** у каждой истории с меткой 🟢 (из `09-mvp.md`) есть AC. Расположение AC — в той же структуре, что выбрана в задаче 0.1 (рядом с историей).
- **Приоритет внутри задачи:** начать с домена T (Decision Process) — это ядро продукта и наиболее нетривиальные AC.

## Блок 2. Модель данных

### 2.1. Спроектировать доменную модель (ERD)

- **Зачем:** сущности определены в `glossary.md` и `07-decision-process.md`, но связи между ними, кардинальности и ключевые атрибуты не описаны. Без ERD разработка начнётся с разных интерпретаций схемы.
- **Входные данные:** сущности из `glossary.md` (User, Group, Project, ProjectLink, Task, Workflow, Status, Transition, Assignment, AssigneeRole, Solution, DecisionCriteria, TaskDecision, Comment, Attachment, TaskLink, Watcher, AuditLog).
- **DoD:** текстовый ERD или таблица сущностей с атрибутами, типами и связями. Отдельно — обоснование нетривиальных решений (например, как хранится `personal_status` каждого Assignment, как связаны Solution и TaskDecision).
- **Особое внимание:** механика Assignment (один Task → много Assignment, у каждого свой статус); Solution → TaskDecision (M:N через accepted_solution_ids[]); soft-delete (поле `deleted_at` у Task и Project).

### 2.2. Описать схему воркфлоу

- **Зачем:** воркфлоу настраивается на уровне проекта. Нужно зафиксировать, как хранятся статусы, допустимые переходы и права на них — это нетривиальная схема.
- **DoD:** описание таблиц Workflow, Status, Transition (с полями); правила: какой пользователь может делать какой переход; как воркфлоу связан с Assignment (персональный статус исполнителя).

### 2.3. Описать модель прав доступа

- **Зачем:** есть роли на уровне проекта (из `02-roles.md`) и роли внутри задачи (AssigneeRole из `04-multi-assignee.md`). Как они сочетаются — нигде не описано.
- **DoD:** матрица: роль (в проекте + в задаче) × действие → разрешено / запрещено. Минимальный набор действий: создать задачу, назначить исполнителя, перевести статус, подать Solution, вынести Decision, изменить видимость проекта, удалить задачу/проект.

## Блок 3. Диаграммы состояний и последовательностей

### 3.1. State-диаграмма задачи с мульти-исполнителями

- **Зачем:** текстовое описание состояний в `07-decision-process.md` есть, но не покрывает граничные случаи: что происходит, если добавить исполнителя в уже закрытую задачу, если отозвать последний неподанный Solution и т.д.
- **DoD:** формальная state-диаграмма (текстом в формате Mermaid или PlantUML) с переходами и условиями. Граничные случаи задокументированы.

### 3.2. Sequence-диаграмма MVP-сценария S2 (Decision Process)

- **Зачем:** S2 — самый сложный сценарий. Последовательность действий акторов, запросов к БД и уведомлений должна быть явной до начала разработки.
- **DoD:** sequence-диаграмма сценария «Менеджер создаёт задачу → назначает исполнителей → каждый подаёт Solution → Decision-maker выносит Decision» с указанием акторов, вызовов и уведомлений.

### 3.3. Схема уведомлений

- **Зачем:** решение по каналам зафиксировано в `10-nfr.md`, но таблица «событие → получатели → канал» нигде не составлена.
- **DoD:** таблица: событие (назначен, прокомментировали, Solution submitted, Decision made и т.д.) × получатели × канал (in-app / email). Отдельно — правила подписки (auto-watchers: все исполнители + автор задачи).

## Блок 4. API-контракты

### 4.1. Описать REST API для MVP-сценариев

- **Зачем:** внутренний API для фронтенда. Без контракта фронтенд и бэкенд будут согласовывать схему в процессе разработки, что замедляет работу.
- **Scope:** только эндпоинты, нужные для 5 MVP-сценариев из `09-mvp.md`.
- **DoD:** для каждого эндпоинта: метод, путь, параметры запроса, схема тела, схема ответа, коды ошибок. Формат: OpenAPI 3.0 YAML или структурированный Markdown.
- **Особое внимание:** эндпоинты для Submit Solution, Create Decision, Request Revision — нетривиальные тела запросов.
