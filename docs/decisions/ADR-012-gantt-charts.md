# ADR-012: Диаграммы Ганта — именованные кросс-проектные таймлайны

**Статус:** Принято  
**Дата:** 2026-05-09

## Контекст

Нужен инструмент планирования на временной шкале поверх проектов. В отличие от Board/Backlog/EpicTree (View внутри проекта), Гант — глобальный объект: охватывает задачи из любых проектов, создаётся независимо от проекта. Таймлайнов может быть несколько, у каждого своё название и набор задач.

## Решение

### Модель данных

**`GanttChart`** — именованная диаграмма:
- `owner_id`, `name`, `description`, `settings` (JSONB, зарезервировано для v2), `position`
- Не привязана к проекту

**`GanttChartTask`** — явно добавленные корневые задачи:
- `gantt_id`, `task_id`, `position`
- Дочерние задачи в таблицу **не записываются** — подтягиваются рекурсивно при запросе

**Task** получил поля: `start_date` (Date), `duration_days` (int, опционально)

### Логика дат (приоритет по убыванию)

| Поле | Применение |
|------|-----------|
| `start_date` | Начало полоски; при отсутствии используется `created_at` |
| `duration_days` | Конец = start + duration (приоритет 1) |
| `due_date` | Явная дата окончания (приоритет 2) |
| max(end дочерних) | Рекурсивно вычисляется, если нет `due_date` (приоритет 3) |
| start + 1 день | Fallback при отсутствии всего остального |

### API

```
GET    /gantt              — список всех диаграмм
POST   /gantt              — создать
PATCH  /gantt/{id}         — обновить название/описание/настройки
DELETE /gantt/{id}
GET    /gantt/{id}/tasks   — все задачи (корневые + потомки через рекурсивный CTE)
POST   /gantt/{id}/tasks   — добавить задачу по task_id
DELETE /gantt/{id}/tasks/{task_id}
```

Отдельный кросс-проектный эндпоинт `GET /tasks` (без project_id) возвращает задачи из всех доступных проектов пользователя.

### Фронтенд

**Страницы:**
- `/timeline` — список всех диаграмм (`GanttListPage`)
- `/timeline/:ganttId` — конкретная диаграмма (`GanttPage`)
- `/timeline/:ganttId/settings` — название, описание, удаление

**GanttChart компонент** (обёртка над `gantt-task-react`):
- Кастомный `TaskListHeader` и `TaskListTable` (стабильная ссылка через `useMemo([], [])` + `useRef` для данных) — список в стиле EpicTreeView: отступы по глубине, иконка типа, ключ-ссылка на задачу, заголовок
- Клик на строку списка → `TaskDetail` как flex-сосед (460px)
- Клик на ключ задачи → полная страница задачи (`/tasks/:key`)
- Колонки на временной шкале разворачиваются/схлопываются через кнопку
- `columnWidth` вычисляется динамически: `containerWidth - listWidth - 16px (VerticalScroll)`, делённый на количество периодов + 6 колонок буфера
- `containerWidth` замеряется через `ResizeObserver` на flex-контейнере в GanttPage (не внутри самого GanttChart wrapper с `overflow:hidden` — это давало некорректную ширину при показе сайдбара)
- Ширина списка изменяется drag-to-resize (180–600px)
- Навигация по шкале: ← / Today / → (±4 нед или ±3 мес в зависимости от zoom)
- Zoom: Week / Month

**Добавление задач:**
- Поиск по ключу задачи (DEMO-4) в поле AddTaskWidget
- Дочерние задачи не нужно добавлять вручную — они подтягиваются через рекурсивный CTE при каждом запросе

**Сайдбар:** секция «Timelines» — первые 5 диаграмм + «All timelines» (аналогично проектам)

### Технические решения

- `gantt-task-react` v0.3.9 (MIT) — выбрана за нативный React, поддержку кастомного TaskListTable и навигацию через `viewDate` проп
- `--legacy-peer-deps` при установке (библиотека декларирует React 18, реально совместима с 19)
- `ganttHeight` не передаётся — высота по содержимому

## Последствия

- Стрелки зависимостей между задачами (sequential-констрейнт) не отрисованы — данные в модели есть, визуализация отложена на v2
- Drag-to-reschedule (перенос дат перетаскиванием) отложен на v2
- Критический путь, baselines, resource view — вне скоупа
- Кросс-проектная область видимости означает, что фильтрация по правам доступа реализована на уровне `list_tasks_global` (только проекты, в которых состоит пользователь)
