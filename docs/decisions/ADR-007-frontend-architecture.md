# ADR-007: Архитектура фронтенда — стек, сайдбар, цветовые токены

**Статус:** Принято  
**Дата:** 2026-04-29

## Контекст

Этап 4 запускает фронтенд. Дизайнер подготовил интерактивный прототип (`design/prototype/`) на React 18 + Babel (без сборки). Прототип использует кастомные CSS-переменные, тёмный сайдбар, все ключевые экраны.

Нужно решить:
1. Как перенести цветовые токены в shadcn/ui
2. Какой подход использовать для сайдбара (тёмный кастомный vs shadcn/ui из коробки)

## Решение

### Стек фронтенда

- **Vite** + **React 19** + **TypeScript** — как запланировано в ADR-002
- **Tailwind v4** — нативная поддержка oklch, CSS-переменные без конфигурации в JS
- **shadcn/ui** — компонентная библиотека, добавление через `npx shadcn@latest add`
- **TanStack Query** — серверный стейт и кэширование
- **axios** — HTTP-клиент с interceptor для Bearer-токена
- **React Router v7** — роутинг
- **Vitest** + **React Testing Library** — юнит-тесты компонентов

### Сайдбар: shadcn/ui из коробки, светлый

Прототип использует тёмный сайдбар (`--sidebar-bg: oklch(0.140)`). Решено использовать **shadcn/ui `<Sidebar>` компонент со светлым фоном по умолчанию**.

**Причины:**
- shadcn/ui Sidebar даёт collapse, мобильный drawer и keyboard nav бесплатно
- Тёмный сайдбар требует отдельного набора переменных и кастомной темизации — дополнительный слой без продуктовой ценности на этом этапе
- Светлый сайдбар не хуже — это выбор дизайна, а не компромисс
- Прототип — референс по структуре и контенту навигации, не по цветам сайдбара

**Что берём из прототипа для сайдбара:** структуру (лого, список проектов, nav items, профиль внизу), иконки, состояния active/hover, поведение при переключении проекта.

**Что НЕ берём:** `--sidebar-bg` тёмный токен, кастомный CSS для сайдбара.

### Цветовые токены

Все CSS-переменные из `TaskTrack.html` (`--bg`, `--fg`, `--primary`, `--border` и др.) переносятся в `frontend/src/index.css` в oklch — Tailwind v4 поддерживает это нативно.

Маппинг прототип → shadcn/ui:
```
--bg              → --background
--bg-card         → --card
--bg-muted        → --muted
--fg              → --foreground
--fg-muted        → --muted-foreground
--border          → --border
--primary         → --primary
--primary-fg      → --primary-foreground
```

Тёмная тема: те же переменные внутри `.dark {}` — Tailwind v4 переключает через `data-theme="dark"` или класс `.dark`.

### Роутинг (Phase 4 scope)

```
/                  → redirect → /projects
/projects          → ProjectsOverview
/projects/:id/*    → (Phase 5+)
/dashboard         → (Phase 5+)
```

### AuthProvider

`AUTH_STUB=true` — бэкенд возвращает фиксированного пользователя. AuthProvider в stub-режиме не делает OIDC-запросов, просто выставляет заглушку текущего пользователя. Keycloak подключается в Этапе 6.

## Последствия

- **Сайдбар upgrade**: если понадобится тёмный сайдбар в v2, достаточно добавить переменные `--sidebar-*` в globals.css — механика останется той же
- **shadcn/ui Sidebar**: новый компонент (появился в конце 2024), хорошо документирован, может требовать `SidebarProvider` на уровне layout
- **Tailwind v4**: конфигурация через CSS (`@import "tailwindcss"`), без `tailwind.config.js`
- **openapi-typescript**: `make gen-types` генерирует типы из `/openapi.json` — запускать после изменений в API
