# ADR-009: BoardColumn как независимый слой между воркфлоу и Kanban-бордой

**Статус:** Принято  
**Дата:** 2026-05-03

## Контекст

До FR-001 Kanban-борда рендерила колонки напрямую из статусов дефолтного воркфлоу проекта: одна колонка = один статус. Это работало пока в проекте был один воркфлоу.

После FR-001 появилась задача: несколько воркфлоу в одном проекте (для task, bug, story, epic, decision — разные жизненные циклы). На борде должны видеть задачи всех типов одновременно. Возникли два варианта отображения:

**Вариант A. Прямой рендер по воркфлоу задачи**  
Каждый воркфлоу — свой набор колонок. Борда показывает несколько секций (по одной на тип задачи), или только задачи текущего выбранного воркфлоу.

_Минусы:_ Менеджер не видит весь поток работ в одном месте. Задачи разных типов (bug + task) физически разделены — нет общей картины. Сложнее понять «что сейчас в Review вне зависимости от типа».

**Вариант B. Независимые BoardColumn с маппингом статусов**  
Менеджер задаёт фиксированный набор колонок (например: To Do / In Progress / Review / Done). Каждой колонке сопоставляется один или несколько статусов из любых воркфлоу проекта. Задача попадает в колонку по своему `current_status_id`.

_Минусы:_ Требует настройки маппинга. Если статус не сопоставлен — задача в «Uncategorized».

## Решение

Выбран **Вариант B** (BoardColumn).

Реализация:
- Новые модели: `BoardColumn(project_id, name, position)` и `BoardColumnStatus(board_column_id, status_id, UNIQUE)`.
- Один статус → максимум одна колонка (уникальный `status_id` в `BoardColumnStatus`).
- `GET /projects/{id}/board-columns` возвращает колонки с вложенными `status_ids`.
- TaskBoard рендерит колонки по `boardColumns`, задача помещается в колонку если `current_status_id ∈ column.status_ids`.
- Drag highlighting: только колонки с достижимыми статусами (через `transitions`) подсвечиваются при перетаскивании.
- Задачи без маппинга — в псевдоколонке «Uncategorized» (видны, не теряются).
- Для DEMO-проекта seed создаёт 4 BoardColumn с маппингом на статусы воркфлоу «Базовый».

Дополнительно решено:
- `Workflow.project_id` сделан nullable: системные воркфлоу (`project_id=NULL`) используются как шаблоны.
- `ProjectTaskTypeConfig(project_id, task_type_id, workflow_id)` — переопределение воркфлоу по типу задачи на уровне проекта.
- `get_workflow_for_task_type` фоллбек-цепочка: `ProjectTaskTypeConfig → project is_default workflow`. Системный воркфлоу из `TaskType.default_workflow_id` **не используется** автоматически — только через явный `ProjectTaskTypeConfig`. Причина: системные воркфлоу не видны в `useProjectWorkflows` → статусы не попадают в `allStatuses` → UI показывает задачи без статуса.

## Последствия

**Плюсы:**
- Борда показывает единый поток работ вне зависимости от типа задачи.
- Гибкость: менеджер сам решает, какие статусы каких воркфлоу соответствуют «In Progress» на борде.
- Воркфлоу задачи фиксируется при создании (`Task.workflow_id`) и не меняется — корректная история.

**Ограничения:**
- При создании нового воркфлоу нужно дополнительно настраивать BoardColumn (маппинг статусов). Без маппинга — задачи в «Uncategorized».
- Если одна колонка содержит несколько статусов и возможны переходы в оба — нужен UI выбора целевого статуса при drop (см. ux-debt.md, пункт 6).

**v2:**
- Показывать выбор статуса при drop на мульти-статусную колонку с резолюцией в одном диалоге (ux-debt.md пункт 6).
- При создании нового воркфлоу предлагать авто-маппинг статусов по категории (`initial → первая колонка`, `final → последняя`).
