# ADR-001: Decision Process как механизм закрытия при мульти-назначении

**Статус:** Принято  
**Дата:** 2026-04-25

## Контекст

Ключевая фича продукта — несколько исполнителей на одну задачу, каждый с независимым воркфлоу. Вопрос: как задача закрывается, если у неё несколько исполнителей? Рассматривались варианты:

- «Все закрыли» — автоматически, с агрегацией резолюций
- «Первый успел» — первый закрывший закрывает задачу для всех
- «Decision менеджера» — явное закрытие менеджером после просмотра результатов

## Решение

При одном исполнителе — классический подход (исполнитель переводит в финальный статус → задача закрыта).

При нескольких исполнителях — запускается **Decision Process**:
1. Каждый lead-исполнитель подаёт **Solution** (поданное решение) при завершении своего воркфлоу.
2. **Decision-maker** (по умолчанию автор задачи) просматривает все Solution'ы и выносит **Decision** — принимает один или несколько Solution'ов.
3. Может отправить Solution'ы на доработку с текстовым комментарием (revision cycle, без ограничения по числу раундов).
4. После вынесения Decision задача переходит в статус `decided`.
5. Decision-maker или менеджер явно закрывает задачу (`decided → closed`).

**Терминальное закрытие одним исполнителем не допускается** — его резолюция закрывает только его часть.

Критерии принятия решения (DecisionCriteria) задаются на уровне задачи, блокируются после первого submitted Solution. Автоматизированный отбор — v2.

Делегирование роли Decision-maker → v2.

## Последствия

- Требует нетривиальной state machine для `Task.global_status` (open → in_progress → awaiting_decision → in_revision → decided → closed).
- Race condition при одновременной подаче Solutions решается через `SELECT FOR UPDATE` на строку задачи.
- Роли внутри задачи (lead/reviewer/consultant) влияют на то, кто обязан подавать Solution и когда задача переходит в `awaiting_decision`.
