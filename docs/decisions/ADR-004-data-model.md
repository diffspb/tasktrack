# ADR-004: Ключевые решения по модели данных

**Статус:** Принято  
**Дата:** 2026-04-26

## Решения

### global_status: хранить как поле, не вычислять

`Task.global_status enum` хранится в БД и обновляется бизнес-логикой при каждом изменении Assignment'ов и Solution'ов. Вычисление на лету потребовало бы агрегации по N Assignment'ам на каждый запрос.

Консистентность обеспечивается через `SELECT FOR UPDATE` при переходах: только один поток меняет `global_status` в каждый момент времени.

### Удаление пользователя: деактивация, не физическое удаление

`User.is_active = false`. Физическое удаление не поддерживается. Деактивированный пользователь: не может войти; его имя отображается как «[Деактивирован]» во всех задачах; его Assignment'ы и Solution'ы остаются в истории.

Причина: FK-ссылки из Assignment, Comment, TaskDecision, AuditLog — физическое удаление ломает историю.

### Воркфлоу изменяемый (не иммутабельный)

Менеджер может редактировать воркфлоу даже при наличии активных задач. Если статус удалён или изменён так, что активные Assignment'ы оказываются в несуществующем статусе:
- Такие Assignment'ы автоматически переводятся в **дефолтный статус воркфлоу**.
- Все участники затронутых задач получают in-app уведомление.
- Событие логируется в AuditLog.

Причина: запрет редактирования создаёт UX-тупик («нельзя исправить ошибку в воркфлоу»). Откат к дефолту — предсказуемый и безопасный fallback.

### Оптимистичные блокировки: только для полей задачи

`Task.version` инкрементируется только при изменении полей задачи (заголовок, описание, приоритет и др.). Смена `Assignment.current_status_id` — не инкрементирует. Это предотвращает ложные конфликты, когда два исполнителя одновременно двигают свои части задачи.

### accepted_solution_ids как PostgreSQL массив

`TaskDecision.accepted_solution_ids uuid[]` — денормализованное поле без FK-ограничений (PostgreSQL не поддерживает FK на массивы). Проверка существования Solution'ов перед созданием TaskDecision — на уровне приложения.

Причина: TaskDecision иммутабелен после создания; Solution'ы не удаляются физически → риск «битых» ссылок минимален.

### Полнотекстовый поиск

Поля `Task.search_vector` и `Comment.search_vector` типа `tsvector`, конфигурация `russian` (snowball). Обновляются триггером при изменении `title`/`description`/`body`. Индекс: `GIN(search_vector)`.

### Kanban: персонализированная доска

На общей доске проекта каждый участник видит задачу в колонке своего **персонального статуса** (`Assignment.current_status_id`). Для пользователей без Assignment — по `Task.global_status`. Индексы на `(task_id, user_id)` и `(user_id, current_status_id)` обязательны.

### decided → closed: два отдельных статуса

`decided` = Decision вынесен, задача ждёт формального закрытия. Участники могут ознакомиться с Decision. `closed` = явное действие decision-maker'а или менеджера. Переход не автоматический.
