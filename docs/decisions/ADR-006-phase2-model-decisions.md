# ADR-006: Решения по модели данных, принятые в Этапе 2

**Статус:** Принято
**Дата:** 2026-04-27

## Контекст

В ходе ревью Этапа 2 (воркфлоу и резолюции) три места реализации разошлись с описанием в `12-data-model.md`. По каждому нужно было решить: подгонять модель под документ или закрепить упрощение.

Объединено в один ADR, так как все три решения — однородные «упростили на MVP, расширим при необходимости».

## Решение

### `Transition.required_role` — одиночное поле, не список

В `12-data-model.md` переход разрешён _списку_ ролей. Реализация хранит одну роль (`required_role: str | None`).

**Принято:** оставить одиночное поле в MVP. Если появится сценарий «переход доступен и lead, и reviewer» — добавить JSON-поле `allowed_roles` и мигрировать данные.

### `Resolution.is_active` — отложено, `position` — добавлено

В `12-data-model.md` у `Resolution` есть `is_active` (архивирование без удаления) и `position`. Реализация: `position` есть, `is_active` нет.

**Принято:** в MVP резолюции только удаляются, не архивируются. Поле `is_active: bool = True` добавим, если появится потребность скрывать резолюции от пользователя без потери данных.

### `Status.is_default` — означает «начальный статус», требует `category=initial`

Текущая семантика: `is_default=True` → ровно один такой статус на воркфлоу, обязательно с `category=initial`. Это «куда попадает Assignment при создании».

**Принято:** оставить так. Если инвариант «ровно один initial-статус как default» станет жёстким, можно упростить до одного поля `category` без `is_default`.

## Последствия

- Все три решения откатываемы: добавить недостающее поле и мигрировать данные не требует ломающих изменений API.
- Документ `12-data-model.md` остаётся источником правды по целевой модели; разница между ним и реализацией Этапа 2 описана здесь.
- Бэкенд-техдолг (APScheduler, Keycloak/JWT, непокрытые тесты) и UX-долг фронтенда вынесены из ADR в [`tech-debt.md`](../tech-debt.md) и [`ux-debt.md`](../ux-debt.md) соответственно.
