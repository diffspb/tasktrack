# ADR-003: Система уведомлений — отказ от email

**Статус:** Принято  
**Дата:** 2026-04-26

## Контекст

В ранних итерациях email рассматривался как основной канал уведомлений для событий Decision Process (awaiting_decision, revision, Decision made). Email — полноценная подсистема: SMTP, баунсы, шаблоны, антиспам, ретраи.

Решение по email пересмотрено: никто не читает рабочую почту достаточно оперативно для Decision Process.

Рассматривались варианты:
- Email (SMTP)
- Мессенджер (Telegram, Slack и др.)
- In-app только

## Решение

**Email убран полностью и навсегда** из MVP и из планов. Не реализовывать.

**MVP:** только badge (счётчик непрочитанных) в шапке приложения. Пользователь проверяет задачи самостоятельно.

**Целевой дизайн (v2):**
- Страница уведомлений у каждого пользователя: хронологическая лента событий, кнопка «отметить как прочитано», групповая отметка.
- Доставка через мессенджер (конкретный канал TBD).

## Последствия

- FastAPI `BackgroundTasks` и APScheduler остаются в стеке только для scheduled напоминаний decision-maker'у (in-app уведомление + badge).
- SMTP-провайдер не нужен.
- В v2 потребуется webhook-интеграция с мессенджером и persistent хранилище уведомлений (таблица `Notification` уже в ERD).
- Таблица `14-diagrams.md` «схема уведомлений» описывает события и получателей, но канал доставки в MVP = только in-app badge.
