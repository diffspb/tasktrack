# 08. Подготовка к проектированию: чек-лист задач

Рабочий документ. Цель — довести продуктовую часть до состояния, в котором её можно передать системному аналитику (без UX-материалов: только текстовое ТЗ).

**Исходные предпосылки** (зафиксированы 2026-04-25):
- Внутренний инструмент, **single-tenant**, без биллинга и self-serve-регистрации.
- Внутри инстанса — **много проектов**, между проектами могут быть **связи**.
- **Видимость проектов** настраивается для участников и групп.
- Поставка аналитику — **только текстовое ТЗ**, без wireframes / прототипов.

## Как читать этот документ

Задачи сгруппированы по блокам. Внутри блока — порядок не критичен, между блоками — зависимости (последующий обычно опирается на предыдущий). Каждая задача содержит:
- Заголовок
- **Зачем** — что блокирует / на что влияет
- **Definition of done** — что считается выполнением
- При необходимости — открытые вопросы

## Блок 1. Фундамент: контекст, границы, словарь

Без этого блока остальные задачи будут формулироваться в разных терминах и с разным пониманием рамок.

### 1.1. Сформулировать контекст и границы продукта
- **Зачем:** одна страница, объясняющая, что мы строим, для кого, в каком окружении. Без этого нельзя приоритизировать истории.
- **DoD:** новый файл `00-context.md` или раздел в `README.md`, отвечающий на: для кого продукт, какую главную проблему решает, какой основной user journey, какие границы (single-tenant, multi-project, ...), что **точно не делаем** в обозримой перспективе.

### 1.2. Зафиксировать «Out of scope»
- **Зачем:** список явно отказанных фичей экономит обсуждения и стабилизирует scope.
- **DoD:** в `00-context.md` (или отдельным разделом) есть список «не делаем»: например, биллинг, self-serve-регистрация, мобильные приложения в первой версии и т.д.

### 1.3. Собрать глоссарий (Ubiquitous Language)
- **Зачем:** в текущих файлах термины расходятся: «исполнитель» / «assignee» / «соисполнитель», «решение» (decision) / «решение» (solution) — два разных значения. Аналитик начнёт с того, что переспросит каждое слово.
- **DoD:** новый файл `glossary.md` со списком терминов и точных определений: Task, Epic, Subtask, Project, ProjectLink, Workflow, Status, Transition, Resolution, Assignment, AssigneeRole, Solution, DecisionCriteria, TaskDecision, Group, Visibility, Watcher, Comment, Attachment, Sprint, Board.
- Согласовать термины во всех существующих файлах после фиксации словаря.

### 1.4. Уточнить разделение значений «решение»
- **Зачем:** в `07-decision-process.md` слово «решение» используется и как submission исполнителя, и как итоговый decision. Это путает.
- **DoD:** в глоссарии явно разведены: **Solution** (вариант, поданный исполнителем) и **Decision** (итоговый выбор decision-maker'а). Все файлы приведены к этой паре.

## Блок 2. Расширение продуктовых требований по новым вводным

Эти требования возникли после первого прохода и пока не описаны в документации.

### 2.1. Описать модель «много проектов в одном инстансе»
- **Зачем:** новое требование, его нет в текущих stories.
- **DoD:** в `02-roles.md` или новом файле описано: пользователь принадлежит инстансу, ему могут быть выданы права в ноль и более проектов; роли в разных проектах независимы.

### 2.2. Описать **связи между проектами**
- **Зачем:** требование «проекты могут быть связаны» — нет описания, что это значит.
- **Открытые вопросы (требуют решения):**
  - Виды связей: иерархия (программа/проект), плоские связи (related), зависимости?
  - Допустимы ли связи между задачами из разных проектов? Если да — как с правами?
  - Можно ли двигать задачу из одного проекта в другой?
- **DoD:** новый раздел в `03-user-stories.md` (или отдельный файл) с user stories по cross-project связям + ответы на вопросы выше.

### 2.3. Описать **группы пользователей и видимость**
- **Зачем:** требование «видимость для участников/групп» — нет в текущих stories.
- **Открытые вопросы:**
  - Иерархия групп? Группы плоские или вложенные?
  - Уровни видимости проекта: public (всем в инстансе) / internal (по списку групп) / restricted (по списку людей) / private (только владельцу)?
  - Видимость задачи внутри проекта может отличаться от видимости проекта?
- **DoD:** в `03-user-stories.md` появился блок «R. Группы и видимость» с историями + матрица «уровень видимости → кто что может».

### 2.4. Интегрировать процесс принятия решения в общие истории
- **Зачем:** `07-decision-process.md` описывает уникальный механизм, но в `03-user-stories.md` соответствующих историй нет (они есть только в `04-multi-assignee.md`). Это разрывает массив историй — аналитик при чтении `03` пропустит ключевую механику.
- **Открытые вопросы:**
  - Истории из `04` **переезжают** в `03` (с удалением из `04`) или **дублируются**?
  - Где лучше держать «единственный источник правды» по историям — в `03` или в тематических файлах?
- **DoD:** принято решение по структуре, в `03-user-stories.md` появился блок «S. Процесс принятия решения» со всеми относящимися историями. Если выбран вариант с дублированием — добавлены явные пометки «дубль из `04`». `04` и `07` связаны cross-references.

## Блок 3. Углубление multi-assignee и Decision Process

`07-decision-process.md` отвечает на «что», но не отвечает на «как именно».

### 3.1. Спецификация Solution
- **Зачем:** «исполнитель презентует решение» — что это технически?
- **Открытые вопросы:**
  - Solution — это структурированная сущность (поля: текст, артефакт, метрики) или свободный текст?
  - Версии Solution: можно ли подавать обновления?
  - Артефакты: ссылки, файлы, и то и другое?
- **DoD:** в `07-decision-process.md` — раздел «Структура Solution».

### 3.2. Спецификация DecisionCriteria
- **Зачем:** «критерии могут быть автоматизированы» — это как?
- **Открытые вопросы:**
  - Формат: набор именованных полей с типами (число, выбор)? Свободный текст?
  - Веса критериев?
  - Кто может менять критерии после старта работы?
- **DoD:** в `07-decision-process.md` — раздел «Структура DecisionCriteria», включая случаи "автоматизированный отбор".

### 3.3. Видимость Solution до вынесения вердикта
- **Зачем:** открытый vs слепой режим — продуктовое решение, влияет на UI и права.
- **DoD:** в `07-decision-process.md` зафиксировано: дефолт — открытый/слепой, можно ли менять на уровне задачи/проекта.

### 3.4. Жизненный цикл и статусы при мульти-исполнителях
- **Зачем:** как именно сосуществуют персональный воркфлоу исполнителя и общий «процесс принятия решения»?
- **Открытые вопросы:**
  - Какие глобальные статусы у задачи: open / in_progress / awaiting_submissions / in_review / decision_made / closed?
  - Когда персональный воркфлоу «закрывается» — Solution автоматически submited?
  - Может ли исполнитель отозвать Solution?
- **DoD:** в `07-decision-process.md` — раздел «Состояния задачи и состояния Solution» со state-диаграммой (текстом).

### 3.5. SLA и уведомления процесса принятия решения
- **Зачем:** decision-maker может «забыть» — нужен ли механизм напоминаний / эскалаций?
- **DoD:** решение по: дедлайну на принятие решения, эскалации, уведомлениям всем участникам о статусе процесса.

### 3.6. Поведение при доработке (revision_requested)
- **Зачем:** «отправлено на доработку» — но не описано, как идёт цикл.
- **Открытые вопросы:**
  - У исполнителя сбрасывается персональный воркфлоу?
  - Decision-maker даёт обратную связь — как (комментарий, структурированное поле)?
  - Сколько раз можно отправлять на доработку?
- **DoD:** в `07-decision-process.md` — раздел «Цикл доработки».

### 3.7. Несколько принятых решений (гипотеза)
- **Зачем:** пользователь хочет проверить эту гипотезу — нужно зафиксировать, как это будет работать в первой версии.
- **DoD:** решение: оставляем как опциональный режим / выносим за MVP / описываем как обязательный.

### 3.8. Роли исполнителей (основной/ревьюер/консультант)
- **Зачем:** ответ «нужны» получен, но семантика ролей не описана.
- **Открытые вопросы:**
  - Какие роли существуют в первой версии?
  - Что отличает их по правам? (например, ревьюер не подаёт Solution?)
  - Можно ли менять роль исполнителя в течение задачи?
- **DoD:** в `04-multi-assignee.md` — раздел «Роли исполнителей в задаче» с таблицей.

## Блок 4. Приоритизация и MVP

### 4.1. Промаркировать все user stories
- **Зачем:** в `03-user-stories.md` 67 историй, в `04-multi-assignee.md` ещё 20 — всего ~87, ни одна не приоритизирована. Без этого MVP не сформулировать.
- **DoD:** каждая история помечена тегом: 🟢 MVP / 🟡 v2 / 🔴 не делаем / ✏️ переформулировать. Помечено и решение по новым блокам R, S.

### 4.2. Сформулировать MVP-сценарии (5–10 штук)
- **Зачем:** конкретные end-to-end сценарии — лучшая защита от размытия scope.
- **DoD:** новый файл `09-mvp.md` со списком сценариев в виде «User journey: вход → действие → результат». Минимум: создать проект, создать задачу, назначить одного исполнителя (классический сценарий), назначить нескольких исполнителей (decision process), переход по воркфлоу, закрытие, поиск.

### 4.3. Acceptance criteria для MVP-историй
- **Зачем:** аналитик не сможет проектировать без явных AC. Формат: Given/When/Then или просто чек-лист условий.
- **DoD:** для всех 🟢 MVP историй прописаны AC. Можно прямо в `03-user-stories.md` под каждой историей.

## Блок 5. Нефункциональные требования (NFR)

Без этого блока аналитик будет угадывать масштаб и качество.

### 5.1. Зафиксировать ожидаемый масштаб
- **DoD:** новый файл `10-nfr.md` или раздел: ожидаемое количество пользователей, проектов, задач за год; пиковая нагрузка на чтение/запись.

### 5.2. Производительность и SLA
- **DoD:** ожидаемое время отклика на ключевые операции (открытие задачи, поиск, kanban), доступность (например, 99,5%), окна обслуживания.

### 5.3. Безопасность и аутентификация
- **DoD:** способ аутентификации (локальный / SSO / OAuth — какие провайдеры), требования к паролям, audit-log, чувствительные операции.

### 5.4. Хранение данных и резервное копирование
- **DoD:** требования к бэкапам, RPO/RTO, политика хранения soft-deleted данных, политика архивации проектов.

### 5.5. Соответствие и приватность
- **DoD:** требования по персональным данным (если релевантно), хранение почты, политика логов.

## Блок 7. Дополнительные продуктовые вопросы

Эти вопросы возникли при анализе и пока не решены. Каждый — задача с открытым ответом.

### 7.1. Поведение due date при мульти-исполнителях
- **Решение:** один общий due date на задачу, у всех исполнителей одинаковый. Персональные due date — v2.

### 7.2. Учёт времени при мульти-исполнителях
- **Решение:** логирование времени — на исполнителя. Задача показывает сумму + разбивку по исполнителям. Сравнение оценка/факт — по каждому исполнителю отдельно.

### 7.3. Эпики и процесс принятия решения
- **Решение:** эпик может содержать задачи в Decision Process. Прогресс эпика считается как (закрытых задач) / (всего задач) — Decision Process не меняет логику расчёта.

### 7.4. Импорт/экспорт в MVP
- **Решение:** CSV-экспорт — с первого дня (🟢 MVP). Импорт из Jira/Trello — не делаем (🔴).

### 7.5. Шаблоны задач
- **Решение:** только клонирование существующей задачи. Выделенная система шаблонов (предзаполненные поля по типу задачи) — v2.

### 7.6. Уведомления — каналы первой версии
- **Решение:** in-app — все события. Email — только события Decision Process (переход в `awaiting_decision`, отправка на доработку, вынесение Decision). Web push — v2.

### 7.7. Watcher-механика для нескольких исполнителей
- **Решение:** все исполнители (lead, reviewer, consultant) и автор задачи — автоматически watchers. Остальные подписываются вручную.

### 7.8. Аналитика процесса принятия решения
- **Решение:** в MVP — только на уровне задачи (время от `awaiting_decision` до `decided`, количество циклов доработки). Агрегированные отчёты по Decision Process — v2.

## Блок 8. Технологические рамки

Не выбор стека, а ограничения, которые задаёт продукт.

### 8.1. Платформа
- **Решение:** только web, mobile-responsive. PWA и нативные приложения — v2.

### 8.2. Браузеры и язык интерфейса
- **Решение:** Chrome ≥ 120, Firefox ≥ 120, Safari ≥ 17. Только русский язык в v1 (i18n-структура заложена). Зафиксировано в `10-nfr.md`.

### 8.3. Развёртывание
- **Решение:** Docker Compose, on-premise / приватное облако (VPS). Зафиксировано в `10-nfr.md`.

### 8.4. API в первой версии
- **Решение:** REST (JSON). MVP — только внутренний API для фронтенда. Публичный API с версионированием — v2. Зафиксировано в `10-nfr.md`.
