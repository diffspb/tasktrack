# Глоссарий

Единый словарь терминов проекта. **Все остальные документы должны использовать термины отсюда.** Если в обсуждении возник синоним — он попадает либо в столбец «синонимы (не использовать)», либо становится новым отдельным термином.

Каждая запись содержит:
- **EN** — каноническое имя для кода и API
- **RU** — единственное допустимое название в документации и UI
- **Определение** — что это
- **Не путать с** — близкие, но разные сущности
- **Синонимы (не использовать)** — слова, которые встречались в черновиках, но больше не должны употребляться

---

## Сущности и роли

### User / Пользователь
Человек с учётной записью в системе. Принадлежит инстансу. В разных проектах может иметь разные роли.

### Group / Группа пользователей
Именованное объединение пользователей. Используется при настройке прав и видимости.

### Project / Проект
Контейнер для задач. У проекта есть владелец, участники, роли, типы задач, воркфлоу, статусы, резолюции. Между проектами могут быть **связи** (см. ProjectLink).

### ProjectLink / Связь между проектами
Отношение между двумя проектами. Конкретный набор типов связей определяется отдельно (см. задачу 2.2 в `08-pre-design-tasks.md`).

### Task / Задача
Атомарная единица работы. Имеет заголовок, описание, тип, приоритет, проект, исполнителей, статус, резолюцию. Может быть подзадачей или частью эпика.

### Epic / Эпик
Крупная инициатива, объединяющая несколько задач. Имеет агрегированный прогресс по входящим в него задачам.

### Subtask / Подзадача
Задача, у которой есть родительская задача. Прогресс подзадач влияет на прогресс родителя.

### Workflow / Воркфлоу
Граф статусов и допустимых переходов между ними для определённого типа задачи.

### Status / Статус
Положение задачи (или Assignment) в воркфлоу. Например: «Новая», «В работе», «Готово».
**Не путать с:** Resolution.

### Transition / Переход
Допустимое изменение статуса с условиями и правами. Например: переход «В работе → Готово» только для тестировщика и только если указана резолюция.

### Resolution / Резолюция
Категория результата задачи: Done / Won't Fix / Duplicate / Cannot Reproduce / Out of Scope и т.д. Назначается **per Assignment** (на каждом исполнителе своя). Не равна итоговому Decision на задачу.
**Не путать с:** Status, Decision.

### Assignment / Назначение
Запись о том, что конкретный пользователь работает над конкретной задачей. У каждого Assignment свой персональный статус, своя резолюция, опционально — своя AssigneeRole. На задаче может быть несколько Assignment одновременно.
**Не путать с:** Task (одна задача → много Assignment).

### AssigneeRole / Роль исполнителя в задаче
Роль конкретного исполнителя в рамках задачи: основной исполнитель, ревьюер, консультант (точный набор будет определён в блоке 3.8 `08-pre-design-tasks.md`).
**Не путать с:** ProjectRole — общая роль пользователя в проекте.

## Решения и процесс

> ⚠️ Слово «решение» в русском многозначно. Мы строго разводим **Solution** и **Decision** и не используем слово «решение» само по себе.

### Solution / Поданное решение
Вариант, который **исполнитель** представляет в финальном статусе своего воркфлоу. Содержит обоснование/артефакт/измеримые результаты по критериям задачи. На одной задаче от каждого Assignment поступает один Solution (с возможными версиями при доработке).
**Не путать с:** Decision (выбирает другой человек).
**Синонимы (не использовать):** «решение исполнителя», «закрытие исполнителем», просто «решение».

### Decision / Итоговое решение
Финальный выбор decision-maker'а: какой Solution (или несколько) принимается как закрытие задачи. Decision принимает одно лицо (decision-maker), и оно бывает только одно на задачу.
**Не путать с:** Solution.
**Синонимы (не использовать):** «вердикт», «итоговая резолюция», просто «решение».

### DecisionCriteria / Критерии принятия решения
Заранее заданные на задаче критерии, по которым decision-maker выбирает Solution. Могут быть формализуемыми (числовые метрики, выбор из списка) или текстовыми. Формализуемые критерии могут поддерживать автоматическую сортировку Solution.

### DecisionMaker / Лицо, принимающее решение
Пользователь, уполномоченный вынести Decision на задаче. По умолчанию — автор задачи. Может быть делегирован.

### Decision Process / Процесс принятия решения
Этап жизни задачи, когда несколько Solution поданы, decision-maker их рассматривает, может отправить часть на доработку, в итоге выбирает Decision. Запускается только при мульти-исполнителях.

### Submission / Подача решения
Действие исполнителя «подать Solution». Не отдельная сущность, а событие.

### Revision / Доработка
Решение decision-maker'а вернуть один или несколько Solution исполнителям с обратной связью. Затронутые исполнители возвращаются в рабочий воркфлоу.

## Поля задачи и связанное

### Label / Метка
Тег на задаче. Используется для фильтрации и поиска. Управляется на уровне проекта или глобально.

### CustomField / Кастомное поле
Дополнительное поле, привязанное к типу задачи (например, «версия», «компонент», «среда»).

### Comment / Комментарий
Сообщение в обсуждении задачи. Поддерживает форматирование, упоминания, реакции.

### Attachment / Вложение
Файл, прикреплённый к задаче или комментарию.

### TaskLink / Связь между задачами
Отношение между двумя задачами: блокирует, блокируется, дубликат, относится к, родительская/дочерняя.
**Не путать с:** ProjectLink (между проектами).

## Видимость и доступ

### Visibility / Видимость
Уровень доступа к проекту или задаче. Конкретные уровни (public / internal / restricted / private) будут зафиксированы в задаче 2.3 `08-pre-design-tasks.md`.

### Permission / Право доступа
Атомарное разрешение действия (создать, редактировать, удалить, перевести статус, изменить настройки и т.д.). Права собираются в Role.

### Role / Роль
Именованный набор Permission. Бывают:
- **ProjectRole** — общая роль пользователя в проекте (менеджер, разработчик, наблюдатель и т.д.)
- **AssigneeRole** — роль конкретного исполнителя в рамках конкретной задачи (см. выше)

### Watcher / Наблюдатель
Пользователь, подписанный на события задачи без участия в исполнении. Может подписаться/отписаться независимо от того, исполнитель он или нет.

## Планирование и представления

### Board / Доска
Визуальное представление задач (Kanban с настраиваемыми колонками).

### Sprint / Спринт
Итерация с датами начала/конца, целью, набором задач (Scrum-режим).

### Backlog / Бэклог
Список задач проекта, ещё не взятых в работу или в спринт.

### View / Представление
Способ отображения списка задач: list, board, calendar, gantt.

### SavedFilter / Сохранённый фильтр
Именованный набор условий поиска. Может быть приватным или расшаренным с командой.

## Системное

### AuditLog / Аудит-лог
Журнал чувствительных действий (удаление, изменение прав, доступ к ограниченным проектам).

### Notification / Уведомление
Сообщение пользователю о событии. Каналы: in-app, email, push.

### Webhook / Веб-хук
Исходящий HTTP-запрос на внешнюю систему при наступлении события.

### Reporter / Автор задачи
Пользователь, создавший задачу.
**Синонимы (не использовать):** «автор» (без уточнения), «постановщик».

### Stakeholder / Заказчик
Лицо, от имени которого ставится задача и в чьих интересах она выполняется. По умолчанию — DecisionMaker.

## Что было «исполнитель»

В черновиках встречались слова **«исполнитель»**, **«assignee»**, **«соисполнитель»**, иногда **«исполнители»** (в смысле всех вместе). Канонизируем:

| Контекст | Используем |
|----------|------------|
| Один человек, выполняющий часть работы по задаче | **исполнитель** (англ. assignee) |
| Запись о назначении этого человека | **Assignment** (англ.) / **назначение** (рус.) |
| Все назначенные на задачу вместе | **исполнители** (мн.ч.) или **assignees** |

Слово **«соисполнитель»** **не используем** — это лишний синоним.
