# 13. Матрица прав доступа

Документ описывает, кто что может делать в системе. Права определяются двумя измерениями:
1. **ProjectRole** — роль пользователя в проекте: Admin, Manager, Member, Viewer.
2. **AssigneeRole** — роль конкретного исполнителя в задаче: lead, reviewer, consultant.

При проверке прав всегда учитывается ProjectRole. AssigneeRole добавляет специфичные права в контексте задачи, но не может дать права выше, чем позволяет ProjectRole.

## Decision Process: права decision-maker'а

Decision-maker — отдельная роль на задаче, не AssigneeRole. По умолчанию — автор задачи. Может быть делегирован.

| Действие | Decision-maker |
|----------|:--------------:|
| Просматривать все Solution | ✅ |
| Вынести Decision (принять Solution'ы) | ✅ |
| Отправить Solution на доработку (revision) | ✅ |
| Задать DecisionCriteria (до блокировки) | ✅ |
| Делегировать роль decision-maker другому | 🟡 v2 |
| Отменить вынесенный Decision | ❌ (только менеджер переоткрывает задачу) |

---

## Проверка прав decision-maker'а

Decision-maker — отдельная роль на уровне задачи, не ProjectRole и не AssigneeRole. Проверяется через сравнение: `task.decision_maker_id == current_user.id`.

- Право «Вынести Decision» (создать `TaskDecision`) проверяется по этому условию.
- Право «Отправить Solution на доработку» (`revision_requested`) проверяется аналогично.
- Делегирование роли decision-maker'а — v2; в MVP `task.decision_maker_id` не меняется после создания задачи (только менеджер или Admin могут изменить через редактирование задачи).

## Маппировка ролей из `02-roles.md` → ProjectRole

Роли из `02-roles.md` описывают бизнес-роли участников. Маппировка на системные роли:

| Бизнес-роль | Системная роль |
|-------------|---------------|
| Автор (Reporter) | Не отдельная ProjectRole. Reporter = пользователь, у которого `task.reporter_id == current_user.id`. Как правило, имеет ProjectRole Member или выше. |
| Наблюдатель (Watcher) | Не отдельная ProjectRole. Watcher = подписчик на задачу (запись в таблице `Watcher`). Может иметь любую ProjectRole вплоть до Viewer. |
| Стейкхолдер/заказчик | Как правило ProjectRole = Viewer. Получает доступ на чтение без права редактирования. |
| Исполнитель (Assignee) | Связан с AssigneeRole (lead/reviewer/consultant) через таблицу `Assignment`. ProjectRole — обычно Member или выше. |
| Менеджер проекта | ProjectRole = Manager. |
| Администратор | ProjectRole = Admin. |

---

## Правила комбинирования ролей

1. **ProjectRole приоритетен**: права ProjectRole применяются к задаче в целом. AssigneeRole расширяет только специфичные для мульти-исполнения действия.

2. **Наивысшая роль при коллизии**: если пользователь имеет доступ через несколько групп — берётся наивысшая ProjectRole.

3. **Admin обходит ограничения**: Admin проекта может выполнять любые действия независимо от AssigneeRole.

4. **Soft-delete и архивирование**: удалённые (soft-delete) и архивированные сущности не возвращаются в стандартных API-запросах; доступны только через специальные admin-запросы.

5. **Кросс-проектные действия**: для установки связи между задачами из разных проектов требуется read-доступ (Viewer или выше) к обоим проектам.
