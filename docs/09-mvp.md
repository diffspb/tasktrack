# 09. MVP: сценарии и приоритизация

Исследовательская цель: прогнать 2–3 реальных проекта через трекер, проверить гипотезу мульти-исполнителей + Decision Process, собрать обратную связь. Масштаб небольшой, сроки сжатые — поэтому приоритет на ядро, а не на полноту.

---

## Ключевые MVP-сценарии

### S1. Классический флоу (один исполнитель)
Менеджер создаёт проект → создаёт задачу → назначает одного исполнителя → исполнитель проводит задачу через воркфлоу → закрывает с резолюцией.

Зачем: базовый юзабилити-контроль. Если это работает плохо — дальше не идём.

### S2. Мульти-исполнители + Decision Process
Менеджер создаёт задачу → прописывает DecisionCriteria → назначает нескольких исполнителей с ролями → каждый ведёт свой воркфлоу независимо → каждый подаёт Solution → decision-maker просматривает Solutionʼы → выносит Decision → задача закрыта.

Зачем: это ядро продукта. Проверяем, что механика понятна и не вызывает трений.

### S3. Цикл доработки
Decision-maker отправляет один Solution на доработку с текстовым комментарием → исполнитель дорабатывает → повторно подаёт Solution → Decision вынесен.

Зачем: доработка — ожидаемый путь, не исключение. Должен быть первоклассным.

### S4. Видимость проекта
Администратор создаёт проект с уровнем `restricted` → добавляет конкретных пользователей / группу → остальные пользователи инстанса проект не видят.

Зачем: нужна хотя бы базовая изоляция для реальных проектов.

### S5. Связь между проектами
Менеджер устанавливает связь `related` между двумя проектами → пользователи видят её в карточке проекта → устанавливает связь между задачами из разных проектов (при наличии прав в обоих).

Зачем: проверяем, нужна ли эта механика на практике и в каком виде.

## Приоритизация user stories

Легенда: 🟢 MVP / 🟡 v2 / 🔴 не делаем в обозримой перспективе / ✏️ переформулировать

### A. Аутентификация и пространство пользователя
- 🟢 Регистрация по email + пароль
- 🟢 Вход через Google OAuth
- 🟡 SSO / SAML
- 🟢 Восстановление пароля
- 🟢 Дашборд «мои задачи»
- 🟢 Настройки уведомлений (базовые)
- 🟢 Профиль: имя, аватар, часовой пояс

### B. Проекты и пространства
- 🟢 Создать проект (название, ключ, описание, владелец)
- 🟢 Пригласить участников, назначить роли
- 🟢 Настроить воркфлоу проекта
- 🟢 Архивировать проект
- 🟢 Список доступных проектов
- 🟢 Независимые роли пользователя в разных проектах

### C. Задачи: создание и управление
- 🟢 Создать задачу (заголовок, описание, тип, приоритет)
- 🟡 Rich-text / Markdown-описание
- 🟢 Прикрепить файлы
- 🟢 Due date и оценка трудозатрат
- 🟢 Связи между задачами (блокирует, дубликат, related)
- 🟢 Редактировать задачу
- 🟢 Soft-delete задачи
- 🟡 Клонировать задачу

### D. Иерархия: эпики, подзадачи
- 🟡 Эпики
- 🟡 Подзадачи

_(Взаимодействие с Decision Process не спроектировано до конца — откладываем)_

### E. Метки, кастомные поля
- 🟢 Метки (теги)
- 🟢 Управление справочником меток
- 🟡 Кастомные поля

### F. Назначение и исполнение
- 🟢 Назначить одного исполнителя
- 🟢 Взять задачу на себя из бэклога
- 🟢 Переназначить задачу
- 🟢 Мульти-исполнители (→ Decision Process) — ядро продукта

### G. Воркфлоу и статусы
- 🟢 Настройка статусов и допустимых переходов
- 🟢 Права на переходы по ролям
- 🟡 Условия на переход (обязательные поля)
- 🟢 Кнопка «следующий статус»

### H. Резолюции и закрытие
- 🟢 Выбор резолюции при закрытии
- 🟢 Настройка набора резолюций на проект
- 🟢 Переоткрытие задачи

### I. Комментирование
- 🟢 Комментарии с форматированием
- 🟢 Упоминания через @
- 🟢 Ответ на конкретный комментарий
- 🟢 Редактировать / удалить свой комментарий
- 🟡 Реакции (👍, ✅)

### J. Поиск и фильтрация
- 🟢 Поиск по тексту (заголовок, описание, комментарии)
- 🟢 Базовые фильтры (проект, исполнитель, метка, статус, дата)
- 🟡 JQL-подобный язык запросов
- 🟡 Сохранённые фильтры
- 🟡 Шаринг фильтров
- 🟢 Виды: список, Kanban
- 🟡 Виды: календарь, диаграмма Ганта

### K. Доски, спринты
- 🟢 Kanban-доска
- 🟡 Спринты и бэклог
- 🟡 Burndown-чарт

### L. Учёт времени
- 🟡 Логирование времени на задачу
- 🟡 Оценка vs факт

### M. История, аудит
- 🟢 История изменений задачи
- 🟡 Аудит-лог по чувствительным действиям

### N. Уведомления
- 🟢 In-app уведомления (все события)
- 🟢 Email для событий Decision Process (Solution requested, revision, Decision made)
- 🟡 Email для всех событий
- 🟡 Дайджест за день/неделю
- 🟢 Подписка/отписка от задачи

### O. Интеграции
- 🟡 Вебхуки
- 🟡 Публичный REST API
- 🟢 CSV-экспорт задач
- 🔴 Импорт из Jira / Trello / GitHub
- 🟡 Интеграция с git (автолинк по ключу задачи)

### P. Права доступа
- 🟢 Роли и наборы разрешений
- 🟢 Скрыть проект для пользователей вне группы

### Q. Аналитика
- 🟡 Базовая отчётность по задачам
- 🟡 Кастомные дашборды

### R. Группы и видимость
- 🟢 Создание групп пользователей
- 🟢 Доступ к проекту по группе
- 🟢 Уровни видимости проекта (public / restricted / private)

### S. Связи между проектами
- 🟢 Связь `related` между проектами
- 🟢 Кросс-проектные связи задач
- 🟡 Иерархические связи (программа → проект)
- 🟡 Перенос задачи между проектами

### T. Процесс принятия решения
- 🟢 Задать DecisionCriteria на задаче
- 🟢 Подать Solution (текст + вложения)
- 🟢 Обновить/отозвать Solution до начала рассмотрения
- 🟢 Decision-maker просматривает все Solutionʼы
- 🟢 Вынести Decision (один или несколько Solution)
- 🟢 Отправить Solution на доработку с комментарием
- 🟢 Статус процесса для наблюдателей
- 🟡 Слепой режим (скрыть Solutionʼы до вынесения Decision)
- 🟡 SLA / напоминание decision-makerʼу (configurable, default 3 дня)
- 🟡 Аналитика Decision Process
