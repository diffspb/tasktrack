# UX-долг (фронтенд)

Открытые UX-проблемы, замеченные при работе с фронтендом, но не блокирующие текущий этап. Закрытые пункты — удалять, история есть в git.

> Для бэкенд-долга — см. [tech-debt.md](./tech-debt.md).

## 1. Нет отображения названия текущего проекта

Когда проект открыт, в шапке и сайдбаре везде написано «TaskTrack» — название самого сервиса. Пользователь не понимает, в каком проекте он находится.

**Варианты:** показать `project.name` в шапке (breadcrumb), выделить активный проект в сайдбаре (жирный + другой цвет), или то и другое.

**Когда делать:** post-MVP (Этап 8 завершён без этого исправления).

## 6. Drag в колонку с несколькими статусами: нет выбора целевого статуса

Когда BoardColumn содержит несколько статусов (пример: «Done» с финальным статусом и «Won't Fix» с другим), и пользователь перетаскивает задачу в эту колонку, система автоматически выбирает **первый доступный** статус по переходам. Пользователь не знает, в какой именно статус попадёт задача.

**Связанная проблема:** если целевой статус финальный (`category=final`), должна открываться форма выбора резолюции — иначе `RESOLUTION_REQUIRED` ошибка. Сейчас task detail открывается сбоку и пользователь должен сам поменять статус там. Это неинтуитивно.

**Решение (делать вместе):**
1. При drop на колонку с ≥2 достижимыми статусами — показывать попап/мини-диалог с выбором статуса.
2. Если выбранный статус финальный — сразу показывать выбор резолюции в том же диалоге.
3. Один flow: drop → статус + (опционально) резолюция → confirm.

**Когда делать:** перед командным пилотом, если в проектах будут колонки с несколькими статусами.

## 7. Гант: стрелки зависимостей — ограничения gantt-task-react

**Единственный цвет стрелок.** `gantt-task-react` принимает один `arrowColor` на весь компонент. Разные типы связей (blocks / depends_on) нельзя покрасить в разные цвета. Для разграничения нужно либо форкнуть библиотеку, либо заменить её рендер на кастомный SVG-слой поверх канваса.

**Стрелки в свёрнутых узлах.** Если родительская задача схлопнута (`hideChildren: true`), её дочерние задачи не попадают в DOM библиотеки — стрелки к/от них не отрисовываются. Нет хука `onExpand` для пересчёта — это ограничение архитектуры gantt-task-react. Правильное решение: при схлопывании «поглощать» зависимости дочерних задач в родителя (как в Jira).

**Когда делать:** при планировании upgrade библиотеки или переходе на кастомный Gantt-рендерер.

## 2. `TaskView.tsx` — мегакомпонент, нет тестов

687 строк, 15+ состояний, вложенные `handleDrop`, `saveTitle`, `saveDesc` inline. Нет ни одного теста. Трудно поддерживать и расширять.

**Решение:** декомпозировать на 5-7 подкомпонентов: `StatusTransition`, `EditTitle`, `DatesBlock`, `ChildTasks`, `LinksSection`. Добавить тесты на ключевые сценарии (переход статуса, редактирование заголовка, открытие дочерних задач).

**Когда делать:** перед активным развитием Decision Process UI, чтобы TaskView мог принять новые блоки без дальнейшего роста.

## 3. Drag-and-drop на Kanban не доступен с клавиатуры

`TaskBoard.tsx:241-248` — только mouse events (`onDragStart`, `onDragOver`, `onDrop`). Пользователи с ограниченными возможностями не могут перемещать карточки. Нарушение WCAG 2.1 Level AA.

**Решение:** добавить keyboard-альтернативу (Space для выбора карточки, стрелки для перемещения между колонками, Enter для подтверждения).

**Когда делать:** перед командным пилотом.

## 8. Нет error boundaries на уровне Layout

Если `AppSidebar`, `Breadcrumbs` или `SearchBar` падают с исключением, весь навигационный слой ломается — пользователь видит белый экран без возможности восстановиться.

**Решение:** обернуть `<Outlet />` в React Error Boundary с fallback «Страница не загрузилась, попробуйте обновить».

**Когда делать:** низкий приоритет, но защита простая — добавить при следующем проходе по Layout.

## 4. Нет точки входа для Decision-maker'а

Когда будет реализован Decision Process (v2), пользователь в роли DM не будет иметь очевидного способа найти задачи, ожидающие его решения.

**Варианты:**
- Бейдж со счётчиком в сайдбаре «Awaiting my decision» рядом с Backlog;
- Глобальный «Inbox» вверху сайдбара со списком задач, требующих внимания.

**Когда делать:** при реализации Decision Process (зависит от восстановления Assignment+Solution).
