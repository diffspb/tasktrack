Открытые UX-проблемы, замеченные при работе с фронтендом, но не блокирующие текущий этап. Закрытые пункты — удалять, история есть в git.

## 1. Нет отображения названия текущего проекта

Когда проект открыт, в шапке и сайдбаре везде написано «TaskTrack» — название самого сервиса. Пользователь не понимает, в каком проекте он находится.

**Варианты:** показать `project.name` в шапке (breadcrumb), выделить активный проект в сайдбаре (жирный + другой цвет), или то и другое.

**Когда делать:** post-MVP (Этап 8 завершён без этого исправления).

## 6. Drag в колонку с несколькими статусами: нет выбора целевого статуса

Когда BoardColumn содержит несколько статусов (пример: «Done» с финальным статусом и «Won't Fix» с другим), и пользователь перетаскивает задачу в эту колонку, система автоматически выбирает **первый доступный** статус по переходам. Пользователь не знает, в какой именно статус попадёт задача.

**Связанная проблема:** если целевой статус финальный (`category=final`), должна открываться форма выбора резолюции — иначе `RESOLUTION_REQUIRED` ошибка. Сейчас task detail открывается сбоку и пользователь должен сам поменять статус там. Это неинтуитивно.

**Решение (делать вместе):**
1. При drop на колонку с ≥2 достижимыми статусами — показывать попап/мини-диалог с выбором статуса.
2. Если выбранный статус финальный — сразу показывать выбор резолюции в том же диалоге.
3. Один flow: drop → статус + (опционально) резолюция → confirm.

**Когда делать:** перед командным пилотом, если в проектах будут колонки с несколькими статусами.

## 4. Нет точки входа для Decision-maker'а

Когда будет реализован Decision Process (v2), пользователь в роли DM не будет иметь очевидного способа найти задачи, ожидающие его решения.

**Варианты:**
- Бейдж со счётчиком в сайдбаре «Awaiting my decision» рядом с Backlog;
- Глобальный «Inbox» вверху сайдбара со списком задач, требующих внимания.

**Когда делать:** при реализации Decision Process (зависит от восстановления Assignment+Solution).
