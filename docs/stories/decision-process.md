## T. Процесс принятия решения (Decision Process)

### Настройка задачи и назначение

- 🟡 Как заказчик задачи, я хочу заранее задать критерии принятия решения (DecisionCriteria), чтобы исполнители понимали, по каким параметрам будет выбран итог.
  - AC:
    - DecisionCriteria задаются как упорядоченный список текстовых пунктов при создании или редактировании задачи.
    - Поле необязательно; задача может не содержать критериев.
    - Критерии блокируются для редактирования, как только хотя бы один lead-исполнитель подал Solution.
    - При попытке редактировать заблокированные критерии — ошибка с пояснением.
    - Автор задачи и менеджер могут задавать и редактировать критерии (до блокировки).

- 🟡 Как менеджер, я хочу назначить нескольких исполнителей с ролями (lead, reviewer, consultant), чтобы запустить мульти-исполнительский воркфлоу.
  - AC:
    - При назначении каждому исполнителю указывается роль: lead, reviewer, consultant (по умолчанию — lead).
    - Только lead-исполнители ведут персональный воркфлоу и подают Solution.
    - reviewer видит все Solution в любое время, но не подаёт своё.
    - consultant не видит Solution до вынесения Decision; может быть упомянут через @.
    - Менеджер может изменить роль исполнителя до момента подачи им Solution; после подачи — смена заблокирована.
    - Задача переходит в `awaiting_decision` только когда все lead-исполнители подали Solution.

- 🟡 Как менеджер, я хочу видеть панель назначений на задаче: все исполнители, их персональные статусы, резолюции, роли.
  - AC:
    - На карточке задачи отображается список всех Assignment с: именем исполнителя, ролью, персональным статусом, резолюцией (если установлена), датой начала работы.
    - Менеджер видит сводный статус задачи: сколько исполнителей завершили, сколько в работе, сколько не начали.
    - Панель обновляется в реальном времени при навигации на страницу.

### Независимый воркфлоу исполнителей

- 🟡 Как один из исполнителей, я хочу видеть свой статус по задаче независимо от статусов других исполнителей.
  - AC:
    - Каждый lead-исполнитель видит свой персональный статус в воркфлоу.
    - Смена персонального статуса не влияет на статусы других исполнителей.
    - В карточке задачи исполнитель видит свой статус выделенным; статусы других — как информацию.
    - Глобальный статус задачи отображается отдельно (open / in_progress / awaiting_decision / in_revision / decided / closed).

- 🟡 Как один из исполнителей, я хочу проводить задачу через свой воркфлоу, не дожидаясь и не блокируя других.
  - AC:
    - Переход в следующий статус воркфлоу доступен исполнителю в любое время, независимо от прогресса других.
    - Кнопка смены статуса всегда активна для исполнителя (если у него есть права на переход).
    - Исполнитель видит статусы других исполнителей в информационном режиме (без возможности изменить).

- 🟡 Как наблюдатель/менеджер, я хочу видеть сводный статус задачи (сколько исполнителей закрыли, сколько в работе, сколько не начали).
  - AC:
    - Сводный статус отображается как агрегированная информация: «3 исполнителя: 1 завершил, 1 в работе, 1 не начал».
    - Глобальный статус задачи меняется автоматически: при старте первого lead'а → `in_progress`, при подаче всех lead-Solution → `awaiting_decision`.

### Подача Solution

- 🟡 Как исполнитель, я хочу подать своё Solution при завершении персонального воркфлоу.
  - AC:
    - Кнопка «Подать Solution» доступна исполнителю (lead) в финальном статусе воркфлоу.
    - Solution содержит: форматированный текст (обязательно) и опциональные вложения (файлы до 20 МБ или URL-ссылки).
    - До подачи Solution находится в статусе `draft`; после нажатия «Подать» — переходит в `submitted`.
    - Подать Solution может только lead-исполнитель; reviewer и consultant — нет.
    - При подаче последнего lead-Solution задача автоматически переходит в `awaiting_decision`.
    - Decision-maker получает in-app + email уведомление при переходе задачи в `awaiting_decision`.

- 🟡 Как исполнитель, я хочу обновлять своё Solution до момента вынесения Decision.
  - AC:
    - Исполнитель может редактировать Solution в статусе `draft` в любое время.
    - Исполнитель может редактировать поданный (`submitted`) Solution, пока decision-maker не начал рассмотрение (пока нет ни одного действия `accepted` / `revision_requested`).
    - История изменений Solution фиксируется в аудит-логе (явное версионирование не хранится).
    - После начала рассмотрения (decision-maker совершил первое действие) — редактирование submitted Solution заблокировано.

- 🟡 Как исполнитель, я хочу отозвать поданный Solution, чтобы доработать перед рассмотрением.
  - AC:
    - Кнопка «Отозвать» доступна, пока Solution в статусе `submitted` и decision-maker не начал рассмотрение.
    - После отзыва Solution возвращается в статус `draft`; задача возвращается из `awaiting_decision` в `in_progress`.
    - Отзыв фиксируется в истории задачи.

### Рассмотрение и вынесение Decision

- 🟡 Как decision-maker, я хочу видеть все поданные Solution с привязкой к критериям, чтобы сравнить их.
  - AC:
    - При переходе задачи в `awaiting_decision` decision-maker видит список всех поданных Solution.
    - Каждый Solution отображается с: именем исполнителя, датой подачи, текстом, вложениями.
    - DecisionCriteria задачи отображаются рядом с Solution'ами для удобства сравнения.
    - Solution'ы отображаются для всех участников задачи (reviewer, decision-maker, lead'ы) в открытом режиме.
    - reviewer видит все Solution всегда; consultant — только после вынесения Decision.

- 🟡 Как decision-maker, я хочу выбрать одно или несколько Solution как итоговое Decision, чтобы закрыть задачу.
  - AC:
    - Decision-maker выбирает один или несколько Solution как принятые.
    - По умолчанию — выбор одного Solution. Мультивыбор доступен, если задача создана с флагом «допускается несколько принятых Solution».
    - После вынесения Decision все lead-исполнители получают in-app + email уведомление.
    - Задача переходит в статус `decided`, затем `closed`.
    - Принятый Solution фиксируется в записи TaskDecision: task_id, decision_maker_id, accepted_solution_ids[], decided_at, note.
    - Отменить вынесенный Decision нельзя (только переоткрыть задачу через менеджера).

- 🟡 Как decision-maker, я хочу отправить одно или несколько Solution на доработку с обратной связью.
  - AC:
    - Decision-maker выбирает один, несколько или все Solution для отправки на доработку.
    - Обязательно указывается текстовый комментарий с обратной связью (не пустой).
    - Выбранные Solution переходят в статус `revision_requested`.
    - Задача переходит в статус `in_revision`.
    - Затронутые исполнители получают in-app + email уведомление с текстом обратной связи.
    - Исполнители, чьи Solution не были отправлены на доработку, остаются в статусе `submitted`.

### Цикл доработки

- 🟡 Как исполнитель, я хочу доработать Solution после запроса на revision и повторно подать.
  - AC:
    - Исполнитель видит комментарий decision-maker'а к своему Solution в статусе `revision_requested`.
    - Исполнитель редактирует Solution и нажимает «Повторно подать».
    - Solution переходит обратно в `submitted`.
    - Когда все Solution из статуса `revision_requested` снова в `submitted` — задача автоматически возвращается в `awaiting_decision`.
    - Количество циклов доработки не ограничено.

### Видимость и контроль процесса

- 🟡 Как наблюдатель, я хочу видеть статус процесса (ожидаются Solution / на рассмотрении / решение принято).
  - AC:
    - На карточке задачи всегда отображается текущий глобальный статус: open / in_progress / awaiting_decision / in_revision / decided / closed.
    - В статусе `awaiting_decision` показывается дата начала рассмотрения.
    - В статусе `decided` / `closed` показывается: принятый Solution (исполнитель, дата), note decision-maker'а.
    - В открытом режиме (MVP по умолчанию) наблюдатели **видят** все поданные Solution'ы. Слепой режим (Solution'ы скрыты до вынесения Decision) — v2.

- 🟡 Как менеджер или decision-maker, я хочу явно закрыть задачу после вынесения Decision (перевести из `decided` в `closed`), чтобы зафиксировать завершение.
  - AC:
    - Действие доступно только когда задача находится в статусе `decided`.
    - Право выполнить это действие имеют: decision-maker задачи и Manager/Admin проекта.
    - После подтверждения задача переходит в статус `closed`.
    - Все участники задачи (Watcher'ы) получают in-app уведомление о закрытии задачи.
    - Событие фиксируется в истории задачи.

- 🟡 Как decision-maker, я хочу получать напоминание через 3 дня, если Decision не вынесен.

- 🟡 Как менеджер, я хочу видеть аналитику Decision Process: среднее время от awaiting_decision до decided, процент revision.

### Управление мульти-исполнителями

- 🟡 Как менеджер, я хочу принудительно завершить задачу, даже если не все исполнители закрыли свои части.
  - AC:
    - Менеджер может закрыть задачу с любым статусом (кроме `decided`) с указанием итоговой резолюции и причины.
    - При принудительном закрытии все незавершённые Assignment получают статус «Закрыто менеджером».
    - Действие фиксируется в истории задачи с именем менеджера и причиной.
    - Уведомление получают все исполнители с незавершёнными Assignment'ами.

- 🟡 Как менеджер, я хочу добавить нового исполнителя в уже идущую задачу.
  - AC:
    - Менеджер выбирает пользователя и роль; создаётся новый Assignment с начальным статусом воркфлоу.
    - Если задача в статусе `awaiting_decision` — менеджер явно подтверждает переоткрытие (задача возвращается в `in_progress`).
    - Если задача уже `decided` / `closed` — добавление нового исполнителя недоступно без переоткрытия задачи.
    - Новый исполнитель получает in-app уведомление о назначении.

- 🟡 Как менеджер, я хочу удалить исполнителя из задачи, если он больше не должен ей заниматься.
  - AC:
    - Менеджер может удалить любого исполнителя (кроме тех, чьи Solution уже в статусе `accepted`).
    - При удалении исполнителя его Assignment закрывается; незавершённый draft-Solution удаляется.
    - Если удалён lead-исполнитель с `submitted` Solution — Solution становится недействительным; задача возвращается в `in_progress`.
    - Действие фиксируется в истории.

## Истории о коммуникации при мульти-исполнении

- 🟡 Как один из исполнителей, я хочу видеть, что делают другие исполнители (их статусы, комментарии), чтобы координироваться.
  - AC:
    - На карточке задачи все lead-исполнители видят список всех Assignment с персональными статусами коллег.
    - reviewer видит все Assignment и все Solution.
    - consultant видит только список исполнителей без их Solution (до вынесения Decision).
    - Комментарии задачи видны всем участникам (исполнители, reviewer, consultant, наблюдатели).

- 🟡 Как пользователь, я хочу на Kanban-доске видеть задачу в колонке, соответствующей моему персональному статусу.
  - AC:
    - Канбан-доска проекта персонализирована: каждый участник видит задачу в колонке своего **персонального статуса** (не глобального статуса задачи).
    - Это означает, что у двух исполнителей одна и та же задача может находиться в разных колонках — каждый видит колонку по своему `Assignment.current_status_id`.
    - Для пользователей без Assignment на задаче — задача отображается в колонке глобального статуса задачи (`Task.global_status`).

- 🟡 Как пользователь, фильтруя «мои задачи в работе», я хочу получать те, где моё назначение в статусе In Progress.
  - AC:
    - Фильтр «Исполнитель: я» + «Статус: In Progress» фильтрует по персональному статусу Assignment пользователя.
    - Задача, где пользователь уже закрыл свою часть, а другие — ещё нет, не попадает в этот фильтр.

## Истории об отчётности при мульти-исполнении

- 🟡 Как менеджер, я хочу отчёты с учётом мульти-назначений: сколько задач на каждом человеке, сколько они закрыли, среднее время на закрытие персональной части.

- 🟡 Как менеджер, я хочу видеть аналитику по комбинациям резолюций исполнителей, чтобы выявлять процессные проблемы.
