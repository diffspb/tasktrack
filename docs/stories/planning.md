# User Stories: Планирование и управление

Домены: D (Epics), E (Labels), J (Search), K (Boards), Q (Analytics), O (Integrations)

---

## D. Иерархия: эпики, подзадачи

- 🟡 Как менеджер, я хочу группировать задачи в эпик (крупная инициатива), чтобы отслеживать прогресс по большой теме, а не по отдельным мелким тикетам.

- 🟡 Как пользователь, я хочу разбить задачу на подзадачи и видеть прогресс родительской через состояние дочерних.

- 🟡 Как менеджер, я хочу видеть прогресс эпика (сколько задач закрыто, сколько в работе, сколько осталось), чтобы коммуницировать статус заказчикам.

_(Взаимодействие эпиков/подзадач с Decision Process не спроектировано до конца — откладываем до v2.)_

## E. Метки, кастомные поля, справочники

- 🟢 Как пользователь, я хочу повесить на задачу метки (теги), чтобы потом искать и фильтровать по ним.
  - AC:
    - Пользователь может добавить метку из существующих меток проекта.
    - На задаче можно выбрать несколько меток.
    - Метки отображаются на карточке задачи и в списке задач.
    - Изменение меток фиксируется в истории задачи.

- 🟢 Как администратор, я хочу управлять справочником меток на уровне проекта, чтобы не было хаоса из синонимов.
  - AC:
    - Администратор/менеджер может создавать, переименовывать и деактивировать метки в настройках проекта.
    - Нельзя удалить метку, которая используется в задачах; можно только деактивировать.
    - Деактивированные метки не предлагаются при добавлении к задаче, но отображаются в существующих задачах.

- 🟡 Как администратор, я хочу добавить кастомные поля к типу задачи (например, «версия», «компонент», «среда»), чтобы фиксировать специфичные для команды атрибуты.

## J. Поиск, фильтрация, представления

- 🟢 Как пользователь, я хочу искать задачи по тексту (заголовок, описание, комментарии).
  - AC:
    - Поиск работает по заголовку, описанию и тексту комментариев.
    - Поиск выполняется с использованием PostgreSQL `tsvector` (полнотекстовый поиск).
    - Результаты отображаются с подсветкой найденного фрагмента.
    - Поиск ограничен задачами из проектов, к которым у пользователя есть доступ.
    - Результаты пагинируются (50 записей на страницу).

- 🟢 Как пользователь, я хочу фильтровать задачи по любой комбинации полей (проект, исполнитель, метка, статус, дата).
  - AC:
    - Доступные фильтры: проект, исполнитель, метка, статус, тип, приоритет, автор, due date (диапазон).
    - Фильтры можно комбинировать; задачи показываются при соответствии всем условиям (AND).
    - Список обновляется при изменении фильтров без перезагрузки страницы.
    - Пустой результат: показывается понятное сообщение «Задачи не найдены».

- 🟡 Как продвинутый пользователь, я хочу язык запросов (типа JQL), чтобы выражать сложные условия.

- 🟡 Как пользователь, я хочу сохранить фильтр как «свой список», чтобы быстро возвращаться к нему.

- 🟡 Как пользователь, я хочу шарить сохранённые фильтры с командой.

- 🟢 Как пользователь, я хочу видеть задачи в представлениях список и Kanban, чтобы выбирать удобный режим.
  - AC:
    - Переключатель «Список / Канбан» доступен на уровне проекта.
    - В представлении «Список»: задачи отображаются построчно с ключом, заголовком, приоритетом, исполнителем, статусом, due date.
    - В представлении «Канбан»: колонки соответствуют статусам воркфлоу; задачи перетаскиваются между колонками для смены статуса.
    - Текущий вид сохраняется в профиле пользователя (не сбрасывается при перезагрузке).

- 🟡 Как пользователь, я хочу видеть задачи в представлениях: календарь, диаграмма Ганта.

## K. Доски, спринты, планирование

- 🟢 Как команда, я хочу Kanban-доску, чтобы видеть поток работы.
  - AC:
    - **MVP (текущее состояние):** колонки борды соответствуют статусам дефолтного воркфлоу проекта. Задача отображается по своему `current_status_id`.
    - Перетаскивание карточки из одной колонки в другую вызывает переход статуса (если переход разрешён воркфлоу).
    - При перетаскивании в колонку без допустимого перехода — карточка возвращается на место.
    - На доске отображаются: ключ задачи, заголовок, приоритет, тип задачи, исполнитель (аватар), метка эпика.
    - Клик по ключу задачи — переход на страницу задачи. Клик по остальной части карточки — открывает панель деталей.

- 🟢 Как менеджер, я хочу настроить колонки Kanban-борды независимо от структуры воркфлоу, чтобы задачи разных типов отображались в одних и тех же логических колонках.
  - AC:
    - Менеджер задаёт набор борд-колонок (название, позиция) в настройках проекта.
    - Каждой колонке сопоставляются один или несколько статусов из воркфлоу проекта (маппинг).
    - Борда рендерится по `board_column`, а не напрямую по `workflow.statuses`.
    - Один статус — ровно одна колонка (нет дублирования карточек).
    - При удалении статуса из воркфлоу — его маппинг на колонку удаляется автоматически.
    - _(реализуется в FR-001 — см. `docs/phase-9-fr001-multi-workflow.md`)_

- 🟡 Как Scrum-команда, я хочу спринты с датами начала/конца, целью спринта, набором задач, чтобы вести итерационное планирование.

- 🟡 Как менеджер, я хочу видеть бэклог проекта с возможностью drag-and-drop приоритизации.

- 🟡 Как менеджер, я хочу burndown-чарт по спринту, чтобы понимать, успеваем ли.

## O. Интеграции и расширяемость

- 🟡 Как администратор, я хочу веб-хуки на события, чтобы интегрироваться с CI, чатами, своими сервисами.

- 🟡 Как разработчик, я хочу публичный REST API, чтобы автоматизировать рутину.

- 🟢 Как пользователь, я хочу экспортировать задачи в CSV, чтобы обрабатывать данные во внешних инструментах.
  - AC:
    - Экспорт доступен из списка задач проекта.
    - Экспортируются все задачи текущего фильтра или все задачи проекта (без фильтра).
    - CSV содержит: ключ, заголовок, статус, приоритет, тип, исполнители, автор, дата создания, due date, метки.
    - Файл формируется и скачивается немедленно (синхронно).

- 🔴 Как пользователь, я хочу импортировать задачи из Jira / Trello / GitHub Issues.

- 🟡 Как команда, я хочу интеграцию с системой контроля версий: при упоминании ключа задачи в коммите — связь автоматически отображается в задаче.

## Q. Аналитика и отчёты

- 🟡 Как менеджер, я хочу отчёты по: количеству задач по статусам/типам, скорости команды (velocity), времени в каждом статусе, распределению по исполнителям, чтобы управлять процессом на основе данных.

- 🟡 Как стейкхолдер, я хочу дашборды с настраиваемыми виджетами, чтобы видеть нужные мне срезы без необходимости лезть в детали.
