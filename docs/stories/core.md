# User Stories: Ядро системы

Домены: A (Auth), B (Projects), C (Tasks), F (Assignment), G (Workflow), H (Resolutions)

---

## A. Аутентификация и пространство пользователя

- 🟢 Как новый пользователь, я хочу зарегистрироваться по email и паролю, чтобы получить доступ к системе.
  - AC:
    - Email должен быть уникальным в системе; повторная регистрация с существующим email возвращает ошибку.
    - Пароль должен содержать не менее 8 символов.
    - После успешной регистрации пользователь автоматически входит в систему.
    - Сессия устанавливается через HTTP-only cookie с TTL 30 дней.

- 🟢 Как новый пользователь, я хочу войти через Google OAuth, чтобы не создавать отдельный пароль.
  - AC:
    - Кнопка «Войти через Google» доступна на экране входа.
    - Если email из Google не существует в системе — создаётся новая учётная запись.
    - Если email из Google уже зарегистрирован локально — учётные записи объединяются (вход под тем же аккаунтом).
    - После успешного OAuth-входа пользователь попадает на дашборд.

- 🟡 Как пользователь, я хочу войти через SSO/SAML, чтобы использовать корпоративные учётные данные.

- 🟢 Как пользователь, я хочу восстановить пароль, если забыл его, чтобы не потерять доступ.
  - AC:
    - На экране входа есть ссылка «Забыли пароль?».
    - На указанный email отправляется письмо со ссылкой для сброса пароля.
    - Ссылка действительна 1 час; повторный запрос инвалидирует предыдущую.
    - После успешной смены пароля все активные сессии завершаются.

- 🟢 Как пользователь, я хочу видеть единый дашборд со «своими» задачами (где я исполнитель, автор, наблюдатель, упомянут), чтобы быстро понять, что от меня ждут.
  - AC:
    - Дашборд отображает задачи, в которых пользователь является исполнителем (Assignment), автором (Reporter), наблюдателем (Watcher), или в которых он упомянут в комментарии.
    - Задачи разбиты на секции: «Мои задачи» (где я исполнитель), «Слежу» (наблюдатель), «Упоминания».
    - Отображаются только задачи из проектов, к которым у пользователя есть доступ.
    - Задачи сортируются по дате последнего обновления (новые сверху).
    - Счётчики в секциях обновляются в реальном времени (при навигации на дашборд).

- 🟢 Как пользователь, я хочу настроить уведомления (базовые), чтобы не утонуть в шуме.
  - AC:
    - Пользователь может отключить каждый тип уведомлений отдельно.
    - Доступные типы: назначение на задачу, смена статуса, новый комментарий, упоминание, события Decision Process.
    - Настройки применяются немедленно.

- 🟢 Как пользователь, я хочу управлять профилем (имя, аватар, часовой пояс), чтобы система корректно показывала время и обращалась ко мне как надо.
  - AC:
    - Пользователь может изменить отображаемое имя, загрузить аватар (JPG/PNG до 2 МБ), выбрать часовой пояс из списка.
    - Все даты и время в UI отображаются в выбранном часовом поясе.
    - Данные сохраняются после нажатия «Сохранить».
    - Аватар отображается во всех местах, где упоминается пользователь (задачи, комментарии, уведомления).

## B. Проекты и пространства

- 🟢 Как администратор, я хочу создать проект с названием, ключом (например, `BACK`), описанием и владельцем, чтобы начать в нём работать.
  - AC:
    - Ключ проекта: 2–10 символов, только латиница и цифры, уникален в инстансе.
    - Название обязательно; описание опционально.
    - Владелец назначается при создании; по умолчанию — создатель.
    - После создания проекта создатель автоматически получает роль Admin в этом проекте.
    - Ключ нельзя изменить после создания.

- 🟢 Как администратор/менеджер, я хочу пригласить участников в проект и назначить им роли, чтобы разграничить права.
  - AC:
    - Пользователь ищется по email или имени из списка пользователей инстанса.
    - При добавлении выбирается роль: Admin, Manager, Member, Viewer.
    - Добавленный пользователь немедленно получает доступ к проекту.
    - Повторное добавление уже участника проекта — ошибка; предлагается изменить роль.

- 🟢 Как менеджер, я хочу настроить для проекта типы задач, воркфлоу, набор статусов и резолюций, чтобы процесс отражал реальность команды.
  - AC:
    - Менеджер может создать воркфлоу: задать статусы и допустимые переходы между ними.
    - Должен существовать хотя бы один начальный и один финальный статус.
    - Менеджер может задать набор резолюций для проекта (добавить/удалить/переименовать).
    - Изменения воркфлоу применяются к новым задачам; существующие задачи остаются в текущем статусе.

- 🟢 Как менеджер, я хочу архивировать неактивный проект, чтобы он не мусорил в списках, но данные сохранялись.
  - AC:
    - Архивированный проект переходит в режим read-only: нельзя создавать задачи, менять статусы, добавлять участников.
    - Архивированный проект не отображается в основном списке проектов; доступен через фильтр «Архив».
    - Разархивировать проект может только Admin или Manager.
    - Данные (задачи, комментарии, история) сохраняются полностью.

- 🟢 Как пользователь, я хочу видеть список доступных мне проектов и быстро между ними переключаться.
  - AC:
    - Список показывает только проекты, к которым у пользователя есть доступ.
    - Отображаются: название, ключ, описание (кратко), дата последней активности.
    - Список поддерживает поиск по названию и ключу.
    - Архивированные проекты скрыты по умолчанию.

- 🟢 Как администратор, я хочу выдавать пользователю доступ к проекту с конкретной ролью, не затрагивая его роли в других проектах, чтобы права в разных проектах были независимы.
  - AC:
    - Роль пользователя в проекте не влияет на его роли в других проектах.
    - Один пользователь может иметь разные роли в разных проектах одновременно.

- 🟢 Как пользователь, я хочу иметь разные роли в разных проектах (например, менеджер в одном и наблюдатель в другом), чтобы система отражала реальную картину моего участия.
  - AC:
    - Права пользователя в интерфейсе динамически применяются в зависимости от текущего проекта.
    - При переключении проекта UI адаптируется: доступные действия соответствуют роли в этом проекте.

## C. Задачи: создание и базовое управление

- 🟢 Как пользователь, я хочу создать задачу с заголовком, описанием, типом, приоритетом и проектом, чтобы зафиксировать работу.
  - AC:
    - Заголовок обязателен; максимальная длина — 500 символов.
    - Тип задачи выбирается из списка типов проекта.
    - Приоритет: Критический, Высокий, Средний, Низкий (по умолчанию — Средний).
    - После создания задача попадает в начальный статус воркфлоу проекта.
    - Задаче автоматически присваивается уникальный номер (ключ проекта + порядковый номер, например `BACK-42`).

- 🟡 Как пользователь, я хочу при создании задачи описывать её в Markdown/rich-text с поддержкой кода, ссылок, изображений, упоминаний, чтобы передать контекст без потерь.

- 🟢 Как автор, я хочу прикрепить к задаче файлы (скриншоты, документы, логи), чтобы исполнитель имел весь контекст.
  - AC:
    - Поддерживаемые форматы: любые файлы до 20 МБ.
    - Максимальный суммарный объём вложений на проект — 500 МБ; при превышении — ошибка с внятным сообщением.
    - Вложения отображаются в разделе задачи; можно скачать и удалить.
    - Удалить вложение может автор вложения или менеджер.

- 🟢 Как автор, я хочу указать срок выполнения (due date) и оценку трудозатрат, чтобы планировать.
  - AC:
    - Due date задаётся как дата (без времени).
    - Оценка трудозатрат задаётся в часах (целое число или десятичная дробь).
    - Оба поля опциональны.
    - Просроченные задачи (due date < сегодня) визуально выделяются в списке и карточке.

- 🟢 Как автор, я хочу задать связи между задачами (блокирует, блокируется, дубликат, related), чтобы видеть зависимости.
  - AC:
    - Доступные типы связей: «блокирует», «блокируется», «дубликат», «related».
    - Связь «блокирует» автоматически создаёт обратную связь «блокируется» у второй задачи.
    - Связь «дубликат» — направленная (задача A является дубликатом задачи B).
    - Для кросс-проектных связей — у пользователя должен быть read-доступ к обоим проектам.
    - Все связи задачи отображаются в отдельном блоке на карточке задачи.

- 🟢 Как пользователь, я хочу редактировать задачу после создания (если у меня есть права).
  - AC:
    - Редактировать задачу могут: автор, исполнители задачи, менеджер, admin.
    - Все изменения фиксируются в истории задачи с указанием пользователя и времени.
    - Нельзя редактировать закрытую задачу (только просматривать) — кроме менеджера и admin.

- 🟢 Как пользователь, я хочу удалять или архивировать ошибочно созданную задачу (с правильными правами).
  - AC:
    - Удаление (soft-delete): поле `deleted_at` проставляется; задача исчезает из всех списков.
    - Физическое удаление — только администратор вручную.
    - Удалить задачу может: автор задачи (если задача в начальном статусе), менеджер, admin.
    - Deleted задачи не отображаются в поиске и фильтрах.

- 🟡 Как пользователь, я хочу копировать/клонировать задачу как шаблон для похожей работы.

## F. Назначение и исполнение

- 🟢 Как менеджер, я хочу назначить задачу на исполнителя, чтобы было понятно, кто за неё отвечает.
  - AC:
    - Менеджер выбирает исполнителя из участников проекта.
    - При назначении можно указать роль: lead, reviewer, consultant (по умолчанию — lead).
    - При назначении создаётся запись Assignment с начальным статусом воркфлоу.
    - Назначенный исполнитель получает in-app уведомление.
    - На задаче отображается список всех исполнителей с их ролями и персональными статусами.

- 🟢 Как менеджер, я хочу назначить нескольких исполнителей с ролями на одну задачу, чтобы запустить мульти-исполнительский воркфлоу.
  - AC:
    - Можно добавить несколько исполнителей с разными ролями (lead, reviewer, consultant).
    - Каждый исполнитель получает свой независимый Assignment с начальным статусом воркфлоу.
    - Задача переходит в статус `in_progress`, когда хотя бы один lead начинает работу.
    - Decision Process активируется только при наличии нескольких lead-исполнителей.

- 🟢 Как пользователь, я хочу взять задачу на себя из бэклога, чтобы менеджеру не приходилось распределять каждую вручную.
  - AC:
    - Пользователь может назначить себя на задачу без исполнителей, если у него роль Member или выше в проекте.
    - При self-assign роль исполнителя устанавливается как lead.
    - Действие фиксируется в истории задачи.

- 🟢 Как исполнитель, я хочу переназначить задачу, если понимаю, что её должен делать другой, чтобы не блокировать процесс.
  - AC:
    - Исполнитель может предложить переназначение; фактическое переназначение подтверждает менеджер.
    - При удалении исполнителя из задачи его Assignment закрывается; незавершённые Solution удаляются.
    - Менеджер может напрямую удалить исполнителя и добавить нового.

## G. Воркфлоу и статусы

- 🟢 Как администратор, я хочу определить набор статусов и допустимые переходы между ними, чтобы воркфлоу отражал реальный процесс.
  - AC:
    - Воркфлоу привязан к проекту; один проект может иметь несколько воркфлоу (для разных типов задач).
    - Статус имеет: название, категорию (open / in_progress / done), цвет.
    - Переходы задаются как пары «из статуса → в статус»; список переходов должен формировать связный граф.
    - Нельзя удалить статус, в котором есть задачи.

- 🟢 Как администратор, я хочу настроить, кто имеет право делать переход, чтобы процесс соблюдался.
  - AC:
    - Для каждого перехода задаётся список ролей, которые могут его совершать.
    - Если роль не задана — переход доступен всем участникам проекта.
    - Права на переход проверяются в реальном времени: пользователь видит только доступные ему переходы.

- 🟡 Как администратор, я хочу повесить условия на переход (например, нельзя закрыть без указания резолюции), чтобы не было «дырявых» закрытий.

- 🟢 Как исполнитель, я хочу одной кнопкой перевести задачу в следующий статус, чтобы это было быстро.
  - AC:
    - На карточке задачи отображается кнопка «Следующий статус» с названием следующего доступного статуса.
    - Если допустимых переходов несколько — показывается выпадающий список.
    - Переход статуса применяется мгновенно без перезагрузки страницы.
    - Смена статуса фиксируется в истории задачи.

- 🟢 Как пользователь, я хочу видеть на задаче только те переходы, которые мне доступны прямо сейчас, чтобы не путаться.
  - AC:
    - Недоступные переходы (из-за роли или условий) не отображаются в UI.
    - Если переходов нет — кнопка «Следующий статус» скрыта.

## H. Резолюции и закрытие

- 🟢 Как исполнитель, я хочу при закрытии задачи выбрать резолюцию (Done / Won't Fix / Duplicate / Cannot Reproduce / Out of Scope), чтобы было понятно, что именно произошло.
  - AC:
    - При переходе в финальный статус воркфлоу выбор резолюции обязателен.
    - Список резолюций настраивается на уровне проекта.
    - Резолюция фиксируется в Assignment (у каждого исполнителя своя резолюция).
    - После выбора резолюции она отображается на карточке задачи рядом с исполнителем.

- 🟢 Как менеджер, я хочу настроить набор доступных резолюций на проект/тип задачи, чтобы команда не закрывала всё подряд как «Done».
  - AC:
    - Менеджер может добавлять, переименовывать и деактивировать резолюции в настройках проекта.
    - Нельзя удалить резолюцию, которая уже используется в закрытых задачах; можно только деактивировать.
    - Деактивированные резолюции не появляются при новых закрытиях, но сохраняются в истории.

- 🟢 Как пользователь, я хочу переоткрыть закрытую задачу, если резолюция оказалась неверной (с записью в историю).
  - AC:
    - Переоткрытие доступно: автору задачи, менеджеру, admin.
    - При переоткрытии задача возвращается в начальный статус воркфлоу (или специальный статус «Переоткрыта», если настроен).
    - Предыдущие резолюции и Assignment'ы сохраняются в истории.
    - Менеджер выбирает, чья часть переоткрывается (при мульти-исполнителях): один, несколько или все.
    - Событие переоткрытия фиксируется в истории с комментарием причины.
