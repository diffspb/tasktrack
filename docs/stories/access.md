# User Stories: Доступ, видимость и группы

Домены: P (Permissions), R (Groups & Visibility), S (Project Links)

---

## P. Права доступа

- 🟢 Как администратор, я хочу настроить роли и наборы разрешений (создание, редактирование, удаление, переход статусов, управление настройками), чтобы разграничить ответственность.
  - AC:
    - В системе доступны четыре встроенные роли: Admin, Manager, Member, Viewer.
    - Каждая роль имеет зафиксированный набор разрешений (кастомизация ролей — v2).
    - Права применяются в реальном времени: смена роли пользователя мгновенно меняет его доступ.
    - Попытка выполнить запрещённое действие возвращает понятную ошибку (403 с описанием, чего не хватает).

- 🟢 Как администратор, я хочу скрыть проект от пользователей вне определённой группы, чтобы поддерживать конфиденциальность.
  - AC:
    - Видимость проекта настраивается как: `public` (видят все в инстансе), `restricted` (только явно добавленные), `private` (только владелец и Admin).
    - Пользователи без доступа не видят проект в списке проектов и не могут открыть его по прямой ссылке (404 или 403).
    - Изменение видимости доступно только Admin проекта.

- 🟢 Как пользователь, я хочу видеть только то, на что у меня есть права, чтобы интерфейс не показывал недоступного.
  - AC:
    - Навигация и кнопки действий рендерятся с учётом прав: недоступные действия не отображаются (не просто задизейблены).
    - API возвращает 403 при обращении к ресурсу без прав, независимо от UI.

## R. Группы пользователей и видимость

- 🟢 Как администратор, я хочу создавать группы пользователей и включать в них участников, чтобы выдавать права не каждому по отдельности.
  - AC:
    - Администратор создаёт группу с именем и опциональным описанием.
    - В группу можно добавить любых пользователей инстанса.
    - Группы плоские (без вложенности).
    - Один пользователь может состоять в нескольких группах.

- 🟢 Как администратор, я хочу назначать доступ к проекту как отдельному пользователю, так и группе.
  - AC:
    - При добавлении доступа к проекту выбирается: пользователь или группа, роль (Admin/Manager/Member/Viewer).
    - Если пользователь добавлен и как индивидуальный участник, и через группу — применяется более высокая роль.
    - При удалении группы из проекта все её участники теряют доступ, полученный через эту группу (если нет индивидуального доступа).

- 🟢 Как администратор, я хочу настроить уровень видимости проекта (`public` / `restricted` / `private`).
  - AC:
    - `public`: все пользователи инстанса видят проект и его задачи в режиме Viewer.
    - `restricted`: доступ только для явно добавленных пользователей и групп.
    - `private`: только владелец проекта и системные Admin'ы.
    - По умолчанию при создании — `restricted`.

- 🟢 Как пользователь, я хочу видеть в списке проектов только те, к которым у меня есть доступ.
  - AC:
    - Список проектов фильтруется на уровне сервера: возвращаются только доступные проекты.
    - Для `public`-проектов доступ есть у всех.
    - Для `restricted`-проектов: только явно добавленные (лично или через группу).
    - Для `private`: только Admin и владелец.

- 🟢 Как пользователь, я хочу понимать, через какую роль или группу мне выдан доступ к проекту.
  - AC:
    - На странице «Мой доступ» в настройках проекта отображается: роль, тип источника (прямое добавление / через группу «Название группы»).
    - Если доступ через несколько групп — отображается наивысшая роль и перечисляются все источники.

## S. Связи между проектами

- 🟢 Как менеджер, я хочу установить связь `related` между двумя проектами, чтобы было видно, что они связаны.
  - AC:
    - Менеджер выбирает проект из списка доступных ему проектов для установки связи `related`.
    - Связь симметрична: оба проекта видят связь на своих страницах.
    - Нельзя установить связь с проектом, к которому нет read-доступа.
    - Один проект может иметь несколько связей.

- 🟢 Как пользователь, я хочу видеть на странице проекта список связанных проектов.
  - AC:
    - На странице проекта отображается блок «Связанные проекты» со списком проектов и типом связи.
    - Если проект связан, но у пользователя нет доступа ко второму проекту — проект в списке показывается как «[Недоступный проект]» (название скрыто).

- 🟢 Как менеджер, я хочу установить связь между задачами из разных проектов (при наличии доступа к обоим).
  - AC:
    - Связь между задачами из разных проектов устанавливается теми же типами, что и внутри проекта (блокирует, related и т.д.).
    - Условие: у пользователя, устанавливающего связь, должен быть read-доступ к обоим проектам.
    - Кросс-проектная связь отображается в блоке связей задачи с указанием проекта второй задачи.
    - Если у пользователя нет доступа к проекту второй задачи — он не видит детали этой задачи, только ключ.
