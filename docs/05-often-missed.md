# 05. Что часто упускают в первой версии

Вещи, которые в MVP кажутся неважными, а потом дорого добавлять задним числом. По каждому пункту — явное решение для первой версии.

---

## Часовые пояса

Всегда хранить время в UTC, отображать в пояс пользователя. Залезть в это потом — означает мигрировать все даты.

**Решение для v1:** ✅ Делаем с первого дня. Все даты хранятся в UTC. Часовой пояс задаётся в профиле пользователя, используется при отображении.

---

## Полнотекстовый поиск

Хотя бы заложить триггеры/индексы (в PostgreSQL — `tsvector`). Без этого поиск по растущему массиву задач быстро становится непригодным.

**Решение для v1:** ✅ PostgreSQL `tsvector` с первого дня. Индексируем: заголовок, описание, комментарии. Elasticsearch — вне рассмотрения для нашего масштаба.

---

## Soft-delete вместо физического удаления

Чтобы можно было восстановить и чтобы аудит не ломался. Также упрощает ссылки на «удалённые» сущности из истории.

**Решение для v1:** ✅ Soft-delete для задач и проектов (поле `deleted_at`). Жёсткое удаление — только вручную администратором. Для комментариев и вложений — не нужен (история ценнее).

---

## Версионирование сущностей и оптимистичные блокировки

Особенно актуально при мульти-назначении, когда несколько людей могут менять задачу одновременно. Поле версии (`version` / `updated_at` с проверкой) спасает от потери изменений.

**Решение для v1:** ✅ Оптимистичные блокировки для задач (поле `version`). При конфликте — показываем ошибку, просим перезагрузить. Для остальных сущностей — не нужно в MVP.

---

## Локализация (i18n)

Даже если запускаешься на одном языке. Хардкод текстов в коде/шаблонах потом извлекать болезненно.

**Решение для v1:** ⚠️ Интерфейс только на русском. Но: строки не хардкодятся в шаблонах — выносятся в файлы локализации с первого дня (i18n-структура есть, второй язык — v2).

---

## Доступность (a11y) интерфейса

Семантическая разметка, ARIA-атрибуты, навигация с клавиатуры. Добавить потом — означает переписывать значимую часть UI.

**Решение для v1:** ⚠️ Базовая семантическая HTML5-разметка. WCAG AA — вне MVP. Внутренний инструмент, целевая аудитория не требует строгого соответствия на старте.

---

## Email как отдельный канал

Со своими сложностями: отписки, баунсы, шаблоны, антиспам, ответы по email обратно в задачу. Не недооценивай его — это полноценная подсистема.

**Решение для v1:** ⚠️ Email — только для событий Decision Process (задача перешла в `awaiting_decision`, Solution отправлен на доработку, Decision вынесен). Все остальные уведомления — только in-app. Ответы по email в задачу — v2.

---

## Идемпотентность операций через API

Особенно для веб-хуков и интеграций. Заложить ключи идемпотентности с самого начала.

**Решение для v1:** ❌ Откладываем. В MVP нет публичного API и вебхуков — поэтому идемпотентность не критична. Заложим при введении публичного API в v2.

---

## Pagination и сортировка как стандарт

Любой список сущностей — с пагинацией (cursor-based лучше offset-based для больших объёмов) и явной сортировкой.

**Решение для v1:** ✅ Пагинация обязательна для всех списков. Тип: offset-based (достаточно для нашего масштаба). Размер страницы по умолчанию: 50. Cursor-based — v2, когда упрёмся в производительность.

---

## Лимиты и rate limiting

Защита от случайных «бомбардировок» API и просто очень больших операций (массовое редактирование 10 000 задач).

**Решение для v1:** ❌ Rate limiting — нет. Внутренний инструмент, доверенное окружение. Если понадобится — добавим на уровне nginx.

---

## Мягкие ограничения на размер вложений

Размер файлов, общий объём на проект — лимиты должны быть с первого дня, иначе позже придётся чистить накопившееся.

**Решение для v1:** ✅ Лимиты с первого дня: 20 МБ на файл, 500 МБ на проект. Превышение — явная ошибка при загрузке.
